# ペルソナ分析 品質レビュープロンプト

> **対象**: `dev/design/persona.md` プロンプトで生成されたペルソナ分析・デザイン方針
> **目的**: 「ペルソナの具体性・リアリティ」「課題・ニーズの深掘り」「デザイン方針との整合性」「セグメント分類・優先順位」「検証可能性・活用設計」の5軸でペルソナ分析を100点満点で評価する。

---

## レビュー前の必須確認事項

レビューを開始する前に、以下をペルソナ分析文書または作成者から確認すること。

| 確認項目 | 確認内容 |
|----------|----------|
| **このプロダクトで解決したい課題は何か？** | 課題を1〜2文で言語化できるか |
| **誰のためのプロダクトか？** | ターゲット顧客が明示されているか |
| **persona.md のどのPhaseまで実施したか？** | 実施Phaseに応じてレビュー深度を調整する |
| **ペルソナ分析の根拠となるデータはあるか？** | ユーザーインタビュー・アンケート・アクセス解析等の有無 |

上記が不明瞭な場合は、レビュー開始前に作成者へ確認すること。

---

## レビュー観点（5項目・各20点・合計100点）

---

### 1. ペルソナの具体性・リアリティ 【配点：20点】

> **目的：** ペルソナが実在するかのように具体的に描かれており、ステレオタイプに陥らず、開発チームが共感できるレベルのリアリティを持っているかを確認する。

- [ ] 年齢・職業・ライフスタイルが具体的な数値・固有名詞を含んで記述されているか
- [ ] 一日の流れ（タイムライン）がリアルに描写されているか
- [ ] ステレオタイプに陥っていないか（「主婦だから〜」「若者だから〜」のような安易な一般化がないか）
- [ ] 実際のユーザーインタビュー・アンケート・行動データなど根拠となる情報源が示されているか
- [ ] 感情・フラストレーション・モチベーションが「生の声」レベルで生々しく描写されているか
- [ ] テクノロジーとの関係（ITリテラシー・使用デバイス・使用シーン）が明確に定義されているか
- [ ] ペルソナの人格・価値観・意思決定基準が読み取れるか
- [ ] 年収帯・生活環境など経済的コンテキストが描写されているか

**採点目安**

| 点数 | 基準 |
|------|------|
| 18〜20 | ペルソナが実在人物のように具体的で、データに裏付けられた感情・行動描写があり、チームが自然に共感できるレベル |
| 14〜17 | おおむね具体的だが、感情描写やデータ根拠の一部が不足している |
| 10〜13 | 基本属性は記載されているが、表面的でステレオタイプ的な記述が多い |
| 0〜9  | ペルソナが抽象的で、具体的な人物像として想像できない |

---

### 2. 課題・ニーズの深掘り 【配点：20点】

> **目的：** 表面的なニーズだけでなく、Jobs to be Done（機能的・感情的・社会的ニーズ）まで掘り下げた分析がされており、ペインポイントが優先順位をつけて整理されているかを確認する。

- [ ] 機能的ニーズ（実際にやりたいこと）が「〜をしたい」形式で具体的に列挙されているか
- [ ] 感情的ニーズ（どう感じたいか）が明示されているか
- [ ] 社会的ニーズ（他者からどう見られたいか）が分析されているか
- [ ] 潜在的ニーズ（ユーザー自身が言語化できていないニーズ）の考察があるか
- [ ] ペインポイントの優先順位づけ（強度評価）がされているか
- [ ] 現在の代替手段（競合・既存の解決策）とその不満点が具体的に分析されているか
- [ ] 課題の頻度・深刻度が定量的に評価されているか（例：強度1-5のスケール）
- [ ] 「よくあるイライラパターン」の具体的なシーンが描写されているか

**採点目安**

| 点数 | 基準 |
|------|------|
| 18〜20 | Jobs to be Doneの3層（機能的・感情的・社会的）が揃い、代替手段の不満と優先順位が定量的に分析されている |
| 14〜17 | おおむね深掘りされているが、潜在ニーズの分析または定量的な優先順位づけが不十分 |
| 10〜13 | 表面的なニーズは記載されているが、感情的・社会的ニーズや代替手段の分析が欠けている |
| 0〜9  | ニーズ分析がほぼなく、ユーザーが何を求めているかが不明確 |

---

### 3. デザイン方針との整合性 【配点：20点】

> **目的：** ペルソナ分析の結果（属性・ニーズ・フラストレーション）からデザイン方針（カラー・フォント・レイアウト・UXパターン）への論理的な導出がなされているかを確認する。

- [ ] カラーパレットの選定根拠がペルソナ属性（年齢・職業・感情トリガー等）から論理的に導出されているか
- [ ] タイポグラフィ選定がペルソナのITリテラシー・年齢層・使用デバイスに適合しているか
- [ ] UXパターンの採用がフラストレーション分析と直結しているか
- [ ] WCAGアクセシビリティ基準（コントラスト比・タッチターゲットサイズ等）が考慮されているか
- [ ] デバイス・利用環境（スマホ中心/PC中心/両方）に合ったレスポンシブ設計方針があるか
- [ ] 感情設計（エモーショナルデザイン）がペルソナの感情的ニーズと連携しているか
- [ ] 情報アーキテクチャ（ファーストビュー配置・情報密度）がペルソナのゴールに基づいているか
- [ ] IBM Carbon Design System等の準拠基準との対応が明記されているか

**採点目安**

| 点数 | 基準 |
|------|------|
| 18〜20 | カラー・フォント・UXパターン・情報設計のすべてがペルソナ分析から論理的に導出され、選定根拠が明記されている |
| 14〜17 | おおむね整合しているが、一部のデザイン決定にペルソナ分析との根拠が不足している |
| 10〜13 | デザイン方針は記載されているが、ペルソナ分析との論理的な接続が弱い |
| 0〜9  | デザイン方針がペルソナ分析と無関係に決定されている、または方針自体がない |

---

### 4. セグメント分類・優先順位 【配点：20点】

> **目的：** ペルソナが適切にセグメント分類され、プライマリ/セカンダリの優先順位が根拠をもって設定されているかを確認する。

- [ ] プライマリペルソナ（最重要ターゲット）が明確に特定され、その根拠が示されているか
- [ ] セカンダリペルソナ（サブターゲット）との違いが明確に説明されているか
- [ ] エッジケースペルソナの設定があり、設計上の考慮事項が記述されているか
- [ ] アンチペルソナ（明確に対象外とするユーザー）が定義されているか
- [ ] ペルソナ間の差異と共通点が比較分析されているか
- [ ] ビジネス上のインパクト（市場規模・収益貢献度）に基づく優先順位づけがあるか
- [ ] 各ペルソナのニーズが矛盾する場合の判断基準が示されているか
- [ ] ペルソナ数が適切か（多すぎて焦点がぼやけていないか、少なすぎて網羅性が低くないか）

**採点目安**

| 点数 | 基準 |
|------|------|
| 18〜20 | プライマリ/セカンダリ/エッジケース/アンチペルソナの区別が明確で、優先順位の根拠とペルソナ間の比較分析がある |
| 14〜17 | おおむね分類されているが、アンチペルソナの定義またはビジネスインパクトに基づく優先順位づけが不足 |
| 10〜13 | ペルソナは複数あるが、優先順位や分類の根拠が不明確 |
| 0〜9  | ペルソナのセグメント分類がなく、優先順位も設定されていない |

---

### 5. 検証可能性・活用設計 【配点：20点】

> **目的：** ペルソナ分析が仮説のまま放置されず、検証計画と開発チームでの実用的な活用方法が設計されているかを確認する。

- [ ] ペルソナの仮説を検証するための具体的な方法（ユーザーインタビュー・A/Bテスト・アンケート等）が示されているか
- [ ] 検証の優先順位とスケジュールが計画されているか
- [ ] 開発チームがペルソナを日常的に参照できる形式（一覧表・カード・ポスター等）になっているか
- [ ] ペルソナに基づくユーザーテストシナリオが設計されているか
- [ ] ペルソナの更新・改訂サイクルの計画（いつ・どのタイミングで見直すか）があるか
- [ ] ペルソナ情報から具体的なユーザーストーリーやシナリオへの展開方法が示されているか
- [ ] デザイン決定サマリーが実装チームへの引き継ぎに十分な情報を含んでいるか
- [ ] KPIやメトリクス（ペルソナ適合度を測る指標）が設定されているか

**採点目安**

| 点数 | 基準 |
|------|------|
| 18〜20 | 検証方法・活用形式・更新計画・テストシナリオ・KPIがすべて揃い、実行可能な計画になっている |
| 14〜17 | おおむね計画されているが、更新サイクルまたはKPI設定が不十分 |
| 10〜13 | 検証方法は一部あるが、チームでの活用設計や更新計画が欠けている |
| 0〜9  | 検証計画・活用設計がなく、ペルソナが作って終わりになるリスクが高い |

---

## 採点方法

### 総合点：100点満点

| 評価 | 点数 | 判定 |
|------|------|------|
| S | 90〜100点 | デザイン実装・開発フェーズへ進む準備完了 |
| A | 80〜89点 | 軽微な補強後にデザイン実装へ進める |
| B | 70〜79点 | 修正推奨（特定観点の深掘りが必要） |
| C | 60〜69点 | 大幅な再検討が必要 |
| D | 59点以下 | ペルソナ分析の根本的な見直しが必要 |

## 重要な注意事項

> **レビュワーは元ファイルを絶対に編集してはならない**
>
> レビューはあくまで「フィードバックの提供」であり、修正作業は作成者自身が行う。
> 改善点は具体的に指摘し、修正案は「提案」として記載すること。

---

## 使用方法

1. このプロンプトと一緒に、レビュー対象のペルソナ分析文書を貼り付ける
2. 「このペルソナ分析をレビューしてください」と依頼する
3. S/A判定であればデザイン決定サマリーをもとに実装フェーズへ進むことを推奨する
4. B以下の場合は改善提案を反映してから再レビューを実施する
