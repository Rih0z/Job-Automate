# CLAUDE.md 生成プロンプト

任意のプロジェクトに対して、Anthropic 公式ベストプラクティスに準拠した `CLAUDE.md` を生成する。

---

## 絶対条件: Anthropic ベストプラクティスに従う

`CLAUDE.md` は Claude Code が**毎セッション自動で全文読み込む**ファイル。1トークンも無駄にしない。以下は交渉不可。

1. **簡潔・具体的** — 一般論・常識・抽象的な指針は書かない。プロジェクト固有の事実・コマンド・落とし穴のみ書く
2. **上限 80 行 / 10KB** — 超えるなら別ファイル化して `@import` または相対リンクで参照
3. **段階的開示（progressive disclosure）** — 詳細ルールは `.claude/rules/*.md` や `docs/` に分離し、CLAUDE.md は索引として機能させる
4. **Markdown のみ** — XML タグ・絵文字・過剰装飾は使わない（ユーザーが明示的に要求した場合を除く）
5. **AI 向けに書く** — 人間向けの導入文・売り文句は書かない（それは README の役割）
6. **古くならない情報を優先** — 進行中タスク・TODO・バージョン番号は書かない（陳腐化する）

---

## 入力

次のいずれか、または複数:
- プロジェクトのディレクトリ（`ls -R`、`tree`、主要ファイルの中身）
- README、`package.json` / `pyproject.toml` / `go.mod` などのマニフェスト
- プロジェクトの口頭説明

---

## 手順

1. **事実収集** — 言語/FW・ディレクトリ責務・実行コマンド・テスト方法・デプロイ方法・既知の落とし穴を抽出
2. **不明点は仮定を明示** — 質問せず仮定を置き、本文内に `> [要確認] ~` として残す。後でユーザーが埋める
3. **テンプレート（下記）に従って生成**
4. **末尾に行数 / バイト数を記載** し上限内であることを自分で確認する
5. **チェックリスト全項目を満たしてから出力**

---

## 出力テンプレート

```markdown
# CLAUDE.md - [プロジェクト名]

> [一行サマリ — 何を、誰のために、どんな技術で作るか]

## 必須手順（任意：作業ごとに守る順序がある場合のみ）

1. [作業開始時のステップ]
2. [...]

## よく使うコマンド

- 起動: `[コマンド]`
- テスト: `[コマンド]`
- ビルド: `[コマンド]`
- [プロジェクト固有のコマンド]

## アーキテクチャ

`[dir1]/` [責務1行] ／ `[dir2]/` [責務1行] ／ ...

## 環境・注意

- [OS依存、特殊パス、絶対NGなど 3〜5項目]

## ルール参照（任意：規模が大きいプロジェクトのみ）

@.claude/rules/[name].md  ← 常時 load する規約
詳細手順: [docs/.../README.md](docs/.../README.md)

## やってはいけないこと

- [過去の失敗から得た 3〜5項目]

---
*[行数] 行 / [KB] KB*
```

不要なセクションは削除する。プロジェクト規模が小さければ「必須手順」「ルール参照」は省く。

---

## ルール分割の判断基準

| 内容 | 配置先 |
|------|-------|
| 全タスク共通（コード品質・テスト方針等） | `.claude/rules/*.md` を作成 → `CLAUDE.md` から `@import` で常時 load |
| 特定タスク時のみ（commit前・Issue起票等） | `.claude/rules/*.md` に YAML frontmatter で `paths:` 指定 → 条件付き load |
| 詳細手順・長いチェックリスト | `docs/` に通常の Markdown → `CLAUDE.md` からはリンクのみ |

## Issue 管理ノウハウ（プロジェクトを問わず適用）

長期運用プロジェクトでは Issue を中心に作業を回す。Issue 管理を採用するプロジェクトでは、**生成する CLAUDE.md に以下のルールを必ず反映**する。GitHub Issues / ファイルベース（`issues/` フォルダ）/ Linear / Jira など、ツールに依らず同じ原則が適用できる。

### 必須ルール

1. **問題を見つけたら即 Issue 化** — タスク実装中に発見した関連バグ・技術的負債・改善案は、**その場で修正せず必ず Issue として記録**する。記録先はプロジェクトの管理方式に合わせる:
   - **ファイルベース**: `issues/open/[ID]-[短いタイトル].md` に新規ファイル作成
   - **GitHub Issues / Linear / Jira**: 該当ツールに新規 Issue 起票
2. **Scope creep 禁止** — 1 Issue = 1 目的。発見した別問題を現タスクに混ぜ込まない。これを破ると差分が肥大化し、レビュー不能・ロールバック不能になる
3. **引継ぎ受領時は与えられた役割のみ実施** — チャット paste でも `.tmp/handoffs/*.md` 等のファイル Read でも、依頼された scope を超えない。気付きは上記ルールに従い別 Issue へ
4. **Issue close 前の自検証** — 受け入れ基準（Acceptance Criteria）充足・テストパス・Lint/型チェックパスを Claude 自身が本文で宣言してから close
5. **一時ファイルとコミット対象を分離** — `.tmp/`（実験・履歴・ハンドオフ）と `.history/`（過去ログ）は `.gitignore` 推奨

### ファイルベース Issue 管理を採用する場合の標準構成

```
issues/
├── open/         未着手・進行中（着手前/中の Issue ファイル）
├── closed/       完了（git mv で open/ → closed/ に移動して履歴保持）
├── handoffs/     エージェント間の引継ぎ（一時的・ .gitignore 任意）
└── TEMPLATE.md   Issue テンプレート（概要・受け入れ基準・関連ファイル・試行履歴）
```

**Issue ライフサイクル**: 発見 → `issues/open/[ID].md` 作成 → 着手 → 完了宣言 → `git mv issues/open/[ID].md issues/closed/[ID].md` で close

**ファイル名規約例**: `2026-05-04-fix-login-timeout.md` または `ISSUE-042-fix-login-timeout.md`（プロジェクトの慣例に合わせる）

### CLAUDE.md への記載例

```markdown
## Issue ワークフロー

- 問題発見時は即 `issues/open/[ID].md` に起票（現タスクに混ぜない）
- 引継ぎ（chat / `.tmp/handoffs/*.md`）受領時は依頼 scope のみ実施
- close は `git mv issues/open/[ID].md issues/closed/[ID].md` で履歴保持
- close 前にテスト結果と受け入れ基準充足を Issue 本文で宣言
- 詳細: @.claude/rules/issue-workflow.md
```

詳細は `.claude/rules/issue-workflow.md` に分離。GitHub Issues 等の外部ツールを使う場合は、上記のファイル操作部分を該当ツールの操作に置き換える。

## セッション・引継ぎ管理ノウハウ（プロジェクトを問わず適用）

長いセッションは**コンテキストが汚染**され、思考が遅く・不正確になる。明示的なリセットと引継ぎ運用を CLAUDE.md に組み込む。

### `/clear` を促すタイミング

Claude は以下の状況でユーザーに `/clear` を提案する:
1. **大きなタスクの完了直後** — Issue close、PR merge、機能リリース後
2. **方向転換時** — 別の Issue / 別のレイヤーに移る時
3. **コンテキスト汚染を自覚した時** — 試行錯誤が長引いた、無関係なファイルを多数読み込んだ
4. **長時間セッション** — 体感で思考が鈍った時、または対話が 1 時間を超えた時

**`/clear` 前に必ずハンドオフファイルを作成する**。書き残さずに clear すると次セッションがゼロから情報収集しなおすことになる。

### 引継ぎ受領のルール

引継ぎを受け取ったら（チャット paste / `.tmp/handoffs/*.md` Read / Issue 本文 などいずれでも）、**それをそのまま引継ぎ資料として扱う**。

| やること | やってはいけないこと |
|---------|------------------|
| 引継ぎ本文をそのまま信頼して着手 | 自分で要約・再解釈・書き直す（情報損失と意図ずれの原因） |
| 不明点があれば**質問する**か `> [要確認]` として残す | 推測で補完して進める |
| 引継ぎに書かれた scope のみ実施 | 周辺の気付きを混ぜ込む（→ 別 Issue 起票） |

### ハンドオフファイルのライフサイクル

引継ぎファイルは**ephemeral（一時的）**。必ず以下のサイクルで管理する:

```
作成（前セッション終端）
  → .tmp/handoffs/[YYYY-MM-DD-task-name].md に保存
  → ユーザーに /clear を促す
  ↓
受領（次セッション冒頭）
  → そのまま信頼して着手（再解釈禁止）
  ↓
タスク完了
  → 受領した引継ぎファイルを削除（必須）
  → 完了の正本記録は commit message / Issue close / PR description に残す
  → handoff ファイルが残ると次セッションが古い情報を読み込む原因になる
```

**完了後に削除しない場合のリスク**: 古いハンドオフが `.tmp/handoffs/` に蓄積し、次のセッションが「これは現タスクの引継ぎか？古いゴミか？」を判断できなくなる。Claude が誤って古い handoff を読んで作業を混乱させる原因になる。

### ハンドオフファイルの構成（推奨）

`.tmp/handoffs/[YYYY-MM-DD-task-name].md` に以下を記載:

```markdown
# 引継ぎ: [タスク名]

## ゴール
[このタスクで達成したいこと 1-2 行]

## 完了したこと
- [...]

## 残課題（次セッションでやること）
1. [最初のアクション — 具体的に]
2. [...]

## 関連ファイル
- [path:line] — [なぜ関連か]

## 既知の落とし穴・注意
- [...]
```

### CLAUDE.md への記載例

```markdown
## セッション管理

- 大きなタスク完了・方向転換時は `.tmp/handoffs/*.md` 作成 → ユーザーに `/clear` を促す
- 引継ぎ受領時は本文をそのまま信頼し、scope を超えない（再解釈・補完禁止）
- **タスク完了時は受領したハンドオフファイルを削除**（正本は commit / Issue / PR に残す）
- 詳細: @.claude/rules/session-handoff.md
```

## 別エージェントレビューのノウハウ（必ず CLAUDE.md に組み込む）

実装とレビューを**同一セッション**で行うと、以下の問題で品質保証が機能しない:

- **コンテキスト汚染** — 実装意図を知っているため好意的解釈してしまう
- **確証バイアス** — 自分が書いたコードの欠陥が見えない
- **採点の甘さ** — 苦労を知っているため無意識にスコアを上げる

### 解決策: コンテキスト分離

レビューエージェントには **事実情報のみ** を渡し、実装の思考過程・会話履歴は渡さない。

| 渡してよい（事実） | 渡してはいけない（実装コンテキスト） |
|----------------|--------------------------------|
| `git diff` / `git log` | 実装の意図・設計判断の理由 |
| 変更ファイルの内容 | 「ここは後で直す予定」等の予告 |
| テスト/CI 設定 | ユーザーとの会話履歴 |
| 評価基準ドキュメント | 試行錯誤の経緯 |

### 起動タイミング

- **commit 直前 / 作業節目** — 実装が一段落した時点で必ず別エージェントレビューを挟む
- **PR 作成前** — テストパスだけでなく、コード品質・テスト網羅性の客観評価を入れる

### 起動方法（プロジェクトに応じて選ぶ）

| 方法 | 内容 |
|------|------|
| `/review-changes` 等のスラッシュコマンド | `Agent` ツールで `subagent_type` を分けて起動。最も自動化されている |
| 別ターミナルで Claude Code を起動 | 完全な情報遮断。手動だが最も確実 |
| GitHub Actions で別 PR レビュー | CI 上で別プロセスとして実行。人手不要 |

### CLAUDE.md への記載例

```markdown
## レビュー（commit 直前・作業節目）

- 実装と同一セッションでレビューしない（コンテキスト汚染防止）
- `/review-changes` で別エージェント起動 → git diff と評価基準のみを渡す
- 詳細な分離ルールは [agents.md](agents.md) 参照
```

別エージェント設計の詳細（コンテキスト遮断ルール・制約と限界）は `agents.md` に分離して記述する。

---

## 含めてはいけない内容

- 一般的な開発常識（「テストを書きましょう」「セキュリティに注意」など）
- ツール自体の使い方の長い説明（公式ドキュメントへリンクすれば十分）
- バージョン番号・進行中タスク・TODO（陳腐化する）
- README に書くべき導入文（CLAUDE.md は AI 向け、README は人間向け）
- 装飾的な見出し階層（h4 以下を多用しない）

---

## 生成後チェックリスト

- [ ] 80 行 / 10KB 以内
- [ ] プロジェクト固有の事実のみで構成されている
- [ ] 全コマンドが実際に存在する（仮の値・推測値が混入していない）
- [ ] 過去の失敗・OS依存・落とし穴が反映されている
- [ ] 詳細ルールは別ファイルに分離され `@import` またはリンクで参照されている
- [ ] 不明点は `> [要確認]` で明示されている
- [ ] 一般論・抽象論・絵文字・XMLタグが含まれていない

## 設計選択: 条文番号方式 vs シンプル方式

| プロジェクト規模 | 推奨フォーマット |
|---------------|---------------|
| 小〜中規模・短期 | シンプルなテンプレート（上記） |
| 大規模・長期運用・複数AI並行作業 | **条文番号方式**（第1条、第2条…）でルールに番号を振り、`CLAUDE.md` は条番号と参照先のみ記載。`.claude/rules/*.md` で本文を管理 |

条文番号方式は「第○条に従い…」と Claude に宣言させることで、ルール遵守の自検証ループを作れる。ただし運用コストが高いので、**規模に見合った形式を選ぶ**こと。

## プロジェクトスキル / スラッシュコマンド管理ノウハウ（プロジェクトを問わず適用）

`.claude/commands/*.md` に置いたスラッシュコマンドは git 管理されチームで共有できる。`~/.claude/commands/*.md` に置けば個人グローバルで使える。

| 配置先 | 共有範囲 | 用途 |
|-------|---------|------|
| `.claude/commands/` | リポジトリ単位（git 共有） | プロジェクト固有のレビュー・自動化コマンド |
| `~/.claude/commands/` | 個人ユーザー全体 | 全プロジェクト共通の自分のお気に入りコマンド |

### 設計原則
- スラッシュコマンドは**自己完結**にする（他リポジトリのパスをハードコードしない）
- 詳細評価基準は `docs/[topic].md` に分離し、コマンド本体からはリンク参照のみ
- エージェント分離が必要なコマンドは `agents.md` 等に分離ルールを記述
- CLAUDE.md には**コマンド一覧表 + 選び方の意思決定フロー**を載せる（コマンド本体は載せない）

## ガバナンス: 新規追加ルールを CLAUDE.md に明記

CLAUDE.md は時間経過で必ず drift する。**「新しい X を追加する時の手順」を CLAUDE.md 内に明記**しておくと、後続の作業者（人間 / AI）が同じ構造を維持できる。

### 記載例

```markdown
## 新しいスキルを追加するルール

1. `.claude/commands/[コマンド名].md` を作成（自己完結に）
2. 詳細基準は `docs/[topic].md` に分離
3. CLAUDE.md のスキル一覧表に追記
4. README.md のテーブルにも追記
5. エージェント分離が必要なら `agents.md` にも追記
```

このセクションを書いておかないと、CLAUDE.md がプロジェクト成長に追従できず形骸化する。

## 意思決定支援パターン

CLAUDE.md には**「何を使えばいいか」の意思決定をサポートする要素**を含める。これがあると Claude もユーザーも判断に迷わない。

### パターン 1: 選び方の decision tree（コードブロックで擬似ツリー）

```markdown
## レビュースキルの選び方

何をレビューしたいか？
├─ 実装コード全体        → /review-implementation
├─ 直近の変更差分        → /review-changes
├─ Skills の品質         → /review-skill
└─ ドキュメント・提案書   → docs/review-*.md を貼り付けて依頼
```

### パターン 2: 前提条件テーブル（あると精度が上がるもの）

```markdown
## /review-implementation を使う前に準備するもの

| 準備                | ファイル例                | なければ                     |
|--------------------|------------------------|------------------------|
| ペルソナ定義         | `dev/design/persona.md` | 推定で評価（注記あり）        |
| 収益モデル           | README に記載          | マネタイズ観点を除外          |
```

### パターン 3: 自検証宣言（条文番号方式で使う）

```markdown
作業開始時の必須手順:
1. 該当ルール（第○条）をユーザーに宣言してから着手
2. 作業後に自検証 → 第○条の他者レビュー
```

宣言を強制することで、Claude がルールを意識せず作業する事態を防ぐ。

## 確実に同レベルの CLAUDE.md を出力するための自己評価

生成後に**自己採点**を必ず行う。1 つでも N の項目があれば書き直す。

| # | 評価項目 | Y/N |
|---|---------|-----|
| 1 | 80 行 / 10KB 以内である |  |
| 2 | プロジェクト固有の事実のみ（一般論ゼロ） |  |
| 3 | 全コマンド・全パスが実在する（推測ゼロ） |  |
| 4 | アーキテクチャがディレクトリ責務 1 行ずつで把握できる |  |
| 5 | Issue 起票ルールが含まれている（or 該当しない理由が明確） |  |
| 6 | エージェント分離レビューの起動タイミングが明記されている |  |
| 7 | `/clear` を促すタイミングと handoff 削除ルールが含まれている |  |
| 8 | スラッシュコマンドの一覧（または該当なしの記述）がある |  |
| 9 | 「新しい○○を追加する手順」のガバナンスセクションがある |  |
| 10 | 詳細ルールは `.claude/rules/*.md` 等に分離され `@import` or リンクのみ |  |
| 11 | 意思決定支援（decision tree / 前提条件テーブル）が少なくとも 1 つある |  |
| 12 | 過去の失敗から得た「やってはいけない」が含まれている |  |
| 13 | 絵文字・XML タグ・装飾的見出し階層が含まれていない |  |
| 14 | 進行中タスク・TODO・バージョン番号など陳腐化する情報がない |  |

## ゴールドスタンダード参照

迷ったら以下を参考にする:

- **索引型**: このリポジトリ（Job-Automate）の `CLAUDE.md` — スキル一覧 + decision tree + 前提条件 + ガバナンス
- **条文番号方式**: AIServer v4 の `CLAUDE.md` — 必須手順 + 条文宣言 + path-scoped rules + 第24条（CLAUDE.md 肥大化防止）

両方とも上記 14 項目をほぼ満たしている。生成物がこのレベルに達しないなら、不足セクションを足してから出力する。
