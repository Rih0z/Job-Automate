# CLAUDE.md 生成プロンプト

任意のプロジェクトに対して、**Anthropic 公式ベストプラクティス**（[code.claude.com/docs/en/best-practices](https://code.claude.com/docs/en/best-practices) 「Write an effective CLAUDE.md」）に厳密準拠した `CLAUDE.md` を生成する。

## 使うタイミング

ユーザーが次のように言った時:
- 「CLAUDE.md を作って」「`/init` の代わりに公式準拠で生成して」
- 「既存の CLAUDE.md を公式ベストプラクティスでレビュー＆書き直して」
- 「このプロジェクトに最適な CLAUDE.md を提案して」
- 「CLAUDE.md が肥大化したので剪定して」
- 「CLAUDE.md を 80 行に収めて」

**使わない時**: プロジェクトのコード自体を書く時、Skills (`.claude/skills/`) を作る時、README を書く時。

---

## YOU MUST: Anthropic 公式の絶対原則

CLAUDE.md は Claude Code が**毎セッション全文 auto-load** するファイル。以下は Anthropic 公式が明示する原則。**1 つでも違反したら出力前に書き直すこと**。

### 公式 Litmus Test（最重要）

> **「この行を消したら Claude がミスをするか？」を 1 行ごとに自問せよ。NO ならカットする。**

これに合格しない行は何があっても削除する。「親切で書いておく」「念のため」は禁止。**肥大化した CLAUDE.md は Claude に実際に無視される**（公式が明言）。

### サイズ閾値（2 段階）

| レベル | 閾値 | 対応 |
|------|------|------|
| **Warning** | 40 行 / 5KB 超過 | 剪定を検討。本当に必要か Litmus Test で再評価 |
| **Fail（出力禁止）** | 80 行 / 10KB 超過 | 必ず分割。`.claude/rules/*.md`・`.claude/skills/`・`docs/` に逃がす |

### 公式推奨の起動・改善サイクル

- 新規プロジェクトでは **`/init` で初期生成** → 公式推奨スターター
- **コードと同じく扱う**: 問題発生時にレビュー、定期的に剪定
- **行動が実際に変わるかで効果を検証**: ルール追加後、Claude の挙動が実際に変わったかを観察
- ルールを追加しても挙動が変わらないなら **CLAUDE.md が長すぎてそのルールが埋もれている** ことを疑う

### 含めるべきもの (Include)

| Include | 理由 |
|---------|------|
| Claude が推測できない Bash コマンド | 起動・テスト・ビルドコマンドなど |
| デフォルトと異なるコード規約 | `import { foo } from 'bar'` を強制等 |
| テスト方法と優先するテストランナー | 「単一テスト優先・全実行回避」など |
| リポジトリの礼儀作法 | ブランチ命名、PR 規約 |
| プロジェクト固有のアーキテクチャ判断 | 「なぜこの構成か」の決定事項 |
| 環境特有の癖 | 必須環境変数、特殊なセットアップ |
| 既知の落とし穴・非自明な挙動 | 「○○すると壊れる」 |

### 含めてはいけないもの (Exclude)

| Exclude | 代わりに |
|--------|---------|
| コードを読めば分かること | 何も書かない |
| 標準的な言語慣習 | 何も書かない |
| 詳細な API ドキュメント | 公式ドキュメントへリンクのみ |
| 頻繁に変わる情報 | 動的に取得 |
| 長文の説明・チュートリアル | `docs/` に分離してリンク |
| ファイル単位の説明 | コードを読ませる |
| 「綺麗なコードを書く」等の自明な助言 | 削除 |
| **時々しか使わない知識・ワークフロー** | **Skills (`.claude/skills/`) に逃がす** |

### 公式ガイドライン

- 形式に決まりはないが**短く、人間が読める**こと
- emphasis（**IMPORTANT** / **YOU MUST**）でルール遵守度を上げてよい
- git に check-in してチームで育てる
- コードと同じく扱う: 問題が起きたらレビュー・剪定
- 行動が実際に変わるかで効果を検証する

## ファイル配置（公式仕様）

| 配置 | 役割 |
|------|------|
| `./CLAUDE.md` | プロジェクトルート・git 共有（**主要な出力先**） |
| `./CLAUDE.local.md` | 個人用・`.gitignore` 必須 |
| `~/.claude/CLAUDE.md` | 全セッション共通の個人設定 |
| 親ディレクトリ | monorepo で自動 merge |
| 子ディレクトリ | そのディレクトリで作業時のみ on-demand load |

`@path/to/file` 構文で他ファイルを import 可能（公式仕様）。

## 生成手順（ゲート方式・全ステップ通過必須）

### Step 1: 事実収集

以下を抽出する。推測・捏造禁止。不明なら `> [要確認]` で残す:
- 言語・FW・ランタイム
- ディレクトリ責務（1 行ずつ）
- 起動・テスト・ビルド・型チェック・lint コマンド（実在するもののみ）
- テスト戦略（単体/統合/E2E、ランナー名）
- デプロイ方法（あれば）
- 環境変数・OS 依存・既知の落とし穴
- git 完全追跡対象の例外があれば（通常 gitignore されるが意図的に追跡しているもの）

**事前 docs 読込宣言フォーマット**（条文番号方式採用時）:
タスク開始時、Claude は関連 docs を最低 1 つ読み、以下を宣言末尾に出力する:
- **完全パス**（例: `docs/16_dev_guides/README.md`）
- **1 文要約**（その docs が何を扱っているか）
- **タスク関連性 1 文**（このタスクとどう関わるか）

これは AIServer v4 第23条の運用パターン。条文方式採用プロジェクトでは生成 CLAUDE.md にこのフォーマットを明記すること。

### Step 2: 候補セクション生成

下記テンプレートに沿って下書き。**各セクションごとに公式 Litmus Test を適用**して残す/削るを決める。

**Litmus Test 適用例**:
- `# Architecture` 直下に `This project uses TypeScript.` → 消したら Claude がミスするか? **NO**（`package.json`/`tsconfig.json` で分かる）→ 削除
- `テスト実行は npm run test:single -- <file>` → 消したら Claude がミスするか? **YES**（推測で `npm test` を実行し全テスト走らせる）→ 残す
- 「コードは綺麗に書きましょう」 → 消したら Claude がミスするか? **NO** → 削除

### Step 3: 段階的開示への分離

CLAUDE.md に直接書かない（= 別ファイルへ逃がす）対象:

| 内容 | 逃がし先 |
|------|---------|
| 全タスク共通の規約（コード品質・テスト方針） | `.claude/rules/[name].md` → `@import` で常時 load |
| 特定タスク時のみのルール（commit 前・Issue 起票等） | `.claude/rules/[name].md` に YAML `paths:` で path-scope |
| 詳細手順・長いチェックリスト | `docs/[topic].md` → リンクのみ |
| **時々しか使わない知識・ワークフロー** | **`.claude/skills/[name]/SKILL.md`** → on-demand load |
| 個人ノート・実験メモ | `CLAUDE.local.md`（gitignore） |

**path-scope rules の YAML frontmatter 例**:

```yaml
---
name: review
description: commit 直前・作業節目に適用するレビュー規約
paths:
  - "**/*.ts"
  - "**/*.tsx"
  - ".claude/commands/**"
---
[ルール本文]
```

`paths:` が match するファイルを Read した時のみ自動 load される。常時 load させたい場合は `paths:` を省略し、`CLAUDE.md` から `@import` で参照する。

### Step 4: テンプレートに従って組み立て

下記「出力テンプレート」に従う。**該当しないセクションは省略**してよいが、省略する場合は理由が明確であること（例: 「テストフレームワークが存在しないため Test セクションを省略」）。

### Step 5: 自己評価（必須ゲート）

下記 26 項目チェックリストを Y/N で全項目評価。**N が 1 つでもあれば出力せずに書き直す**。

### Step 6: サイズ確認（実コマンド必須）

`wc -l <生成 CLAUDE.md パス>` と `wc -c <生成 CLAUDE.md パス>`（PowerShell では `(Get-Content <path>).Count` と `(Get-Item <path>).Length`）を実行し、その実測値を末尾の `*[行数] 行 / [KB] KB*` に転記する。**推定値・概算値の記入は禁止**。80 行 / 10KB を超過していたら出力せず Step 2 に戻って剪定する。

### Step 7: 別エージェントによる公式ベストプラクティス準拠レビュー（必須・削除不可ゲート）

**自己評価で全 Y を取っても、それだけでは出力しない。**生成した CLAUDE.md を**別エージェント**（`Agent` ツール `subagent_type: "general-purpose"`、または別ターミナルの Claude Code セッション）に渡し、Anthropic 公式ベストプラクティス準拠レビューを受ける。

#### レビュアエージェントが必ず実施すること（YOU MUST）

1. **公式ドキュメントを Web から実際に取得**する（記憶・推測でレビューしない）
   - `WebFetch` ツールで `https://code.claude.com/docs/en/best-practices` を取得
   - "Write an effective CLAUDE.md" セクション全文を読む
   - **取得日時とドキュメントの主要原則を引用**してレビュー冒頭で宣言する
2. 取得した公式 Best Practices の Include/Exclude テーブル、Litmus Test、サイズガイダンス、emphasis、`@import` 仕様、ファイル配置（CLAUDE.md / CLAUDE.local.md / 親子）に対し、生成 CLAUDE.md を 1 項目ずつ Y/N 評価
3. `docs/review-skill.md`（プロジェクト内にあれば）の 5 軸 100 点満点採点
4. 引用した公式ドキュメントの該当箇所と生成 CLAUDE.md の該当行を**並べて**示し、根拠を明示

#### レビュアに渡すもの（事実のみ）

- 生成した CLAUDE.md の中身（または絶対パス）
- `docs/review-skill.md`（プロジェクト内にあれば）の絶対パス
- レビュア自身が `WebFetch` で公式ドキュメントを取りに行くよう明示的に指示

#### レビュアに渡さないもの

- 本生成プロンプト
- 公式ドキュメントの引用や要約（レビュアに**自分で取りに行かせる**ことで記憶ベースのレビューを防ぐ）
- 生成過程の試行錯誤・自己評価結果
- ユーザーとの会話履歴

#### 合格基準

**公式 Best Practices 全項目 Y かつ 90 点以上、かつレビュア冒頭で公式ドキュメントの取得日時引用がある**

レビュアが Web から公式取得していない場合は無効。再依頼する。

#### 不合格時

指摘箇所を修正して再レビュー。**3 回失敗したら `issues/open/[YYYY-MM-DD]-claude-md-generation.md` に起票し、当該タスクを中断してユーザーに報告**。

### Step 8: 出力 & ハンドオフ

レビュー合格後、ユーザーに次のことを伝える:
- 生成内容の要約（3 行以内）
- 残した `> [要確認]` 項目があればリスト化
- 関連して作成すべき `.claude/rules/*.md`, `.claude/skills/*/SKILL.md`, `agents.md`, `docs/*.md` の提案
- レビュア合格スコア

### Step 9: `/clear` 実行 + 次セッション継続プロンプト生成（必須）

CLAUDE.md 生成という大タスクが完了したため、コンテキスト汚染防止に**ユーザーに `/clear` の実行を促す**。`/clear` 前に、次セッションがゼロから情報収集しなおさずに済むよう、以下の継続プロンプトを `.tmp/handoffs/[YYYY-MM-DD]-claude-md-generated.md` に保存する:

```markdown
# 引継ぎ: CLAUDE.md 生成完了 → 次セッションで反復改善

## ゴール
生成された CLAUDE.md（`./CLAUDE.md`）を実運用に投入し、Claude の挙動が実際に変わるかを観察して反復剪定する。

## 完了したこと
- `./CLAUDE.md` を Anthropic 公式 Best Practices 準拠で生成（[行数] 行 / [KB] KB）
- 自己評価 21 項目: 全 Y
- 別エージェントレビュー: [スコア]/100 で合格
- 関連ファイル作成提案: [.claude/rules/*.md など]

## 次セッションでやること
1. 生成 CLAUDE.md を読み込んで Claude の挙動が変わるか観察
2. ルールを追加しても挙動が変わらない箇所があれば「埋もれている」と判断し剪定
3. 残した `> [要確認]` 項目をユーザーに確認して埋める
4. 関連ファイル（`.claude/rules/*.md` 等）を実際に作成

## 関連ファイル
- `./CLAUDE.md` — 今回生成したもの
- `dev/claude-md.md` — 生成プロンプト本体（再生成・剪定時に参照）
- `[要確認項目に該当するファイル]`

## 既知の落とし穴・注意
- ハンドオフ受領時は本文をそのまま信頼（再解釈・要約・書き直し禁止）
- 着手後は scope を超えない（気付きは別 Issue へ）
- このハンドオフファイルはタスク完了時に**削除する**
```

ユーザーに次のように伝える:

> CLAUDE.md の生成が完了しました（[スコア]/100 でレビュー合格）。次セッションで反復改善するためのハンドオフを `.tmp/handoffs/[ファイル名].md` に保存しました。コンテキスト汚染防止のため `/clear` を実行してください。次セッション冒頭で「`.tmp/handoffs/[ファイル名].md` を読んで続行」と指示すれば、本タスクの続きから着手できます。

## YOU MUST: レビュー受領サイクル（全成果物に対して必須）

すべての成果物（実装・設計・ドキュメント）は、実行後に**別エージェントから skills を使ったレビューを最低 1 回受ける**。

1. 実装/変更を完了させる
2. 別エージェントに該当 skill でレビューを依頼（実装→`/review-changes`, スキル→`/review-skill`, 提案書→`docs/review-proposal.md` 等）
3. 渡すのは **git diff と評価基準のみ**（実装意図・会話履歴は渡さない）
4. 合格基準（例: 90 点以上）に達するまで修正→再レビューを反復
5. **3 回失敗したら `issues/open/[ID].md` に起票して当該タスクを中断**、ユーザーに報告

詳細: [agents.md](agents.md)

## 必ず取り込むノウハウ（プロジェクトを問わず適用）

以下は Anthropic 公式が直接明記していないが、**実運用で公式原則を実現するための運用ノウハウ**。該当しない場合のみ省略してよい。

### A. Issue 管理

| ルール | 内容 |
|------|------|
| **問題発見即起票** | タスク中に発見した別バグ/改善は現タスクで触らず `issues/open/[ID].md` 等に記録 |
| **1 Issue = 1 目的** | scope creep 禁止。差分肥大とロールバック不能を防ぐ |
| **close 前自検証** | 受け入れ基準・テストパス・lint 通過を本文宣言 |
| **ファイル運用** | `issues/open/` ↔ `issues/closed/`（git mv で移動）、`TEMPLATE.md` を置く |

GitHub Issues / Linear / Jira でも同じ原則を適用。

### B. セッション管理・引継ぎ

| ルール | 内容 |
|------|------|
| **`/clear` を促すタイミング** | 大タスク完了、方向転換、コンテキスト汚染自覚時、長時間セッション後 |
| **`/clear` 前に handoff 作成** | `.tmp/handoffs/[YYYY-MM-DD-task].md` に書いてから clear |
| **引継ぎ受領時は再解釈禁止** | そのまま信頼。不明点は質問または `> [要確認]` で残す |
| **完了時に handoff 削除（必須）** | 正本は commit / Issue / PR。残すと次セッションが古い情報を読み混乱する |

ハンドオフ構成: ゴール／完了したこと／残課題／関連ファイル（path:line）／落とし穴。

### C. 別エージェントレビュー（コンテキスト分離・必須サイクル）

| 渡してよい（事実） | 渡してはいけない（実装コンテキスト） |
|----------------|--------------------------------|
| `git diff` / `git log` | 実装の意図・設計判断の理由 |
| 変更ファイルの内容 | 「ここは後で直す」等の予告 |
| テスト/CI 設定 | ユーザーとの会話履歴 |
| 評価基準ドキュメント（skills） | 試行錯誤の経緯 |

公式が推奨する **Writer/Reviewer パターン**（公式ドキュメント明記）と一致する。起動方法:
- スラッシュコマンド（`/review-changes` 等）で `Agent` ツール経由
- 別ターミナルで Claude Code を別セッション起動
- GitHub Actions など CI 上の別プロセス

#### 必須レビューサイクル（YOU MUST）

**すべての動作・成果物は、実行後に別エージェントから skills を使ったレビューを最低 1 回以上受ける。** 合格するまで反復し、3 回失敗で Issue 化する。

```
[実行] → [別エージェントレビュー (skills 使用)] → 合格？
                                              ├─ Yes → 完了
                                              └─ No  → 修正して再レビュー
                                                         │
                                                         ├─ 2 回目失敗 → 修正して再レビュー
                                                         └─ 3 回目失敗 → Issue 起票（issues/open/[ID].md）
                                                                          → 当該タスクを中断
                                                                          → ユーザーに報告
```

**レビューに使う skills（プロジェクト内のコマンドを優先）**:
- 実装変更 → `/review-changes`（または `docs/review-changes.md`）
- スキル/プロンプト変更 → `/review-skill`（または `docs/review-skill.md`）
- 提案書/ドキュメント変更 → `docs/review-proposal.md` 等
- プロジェクト全体の総合 → `/review-implementation`（または `docs/review-implementation.md`）

**3 回失敗時の Issue 起票テンプレ**:

```markdown
# 3 回レビュー失敗: [タスク名]

## 経緯
- 試行 1: [変更内容] → 落ちた評価軸: [軸名] (理由)
- 試行 2: [変更内容] → 落ちた評価軸: [軸名] (理由)
- 試行 3: [変更内容] → 落ちた評価軸: [軸名] (理由)

## 仮説
[なぜ合格しないかの構造的な問題]

## 次の打ち手案
[評価軸自体の見直し / アプローチ変更 / 専門家相談 など]
```

### CLAUDE.md への記載例（C. に対応）

```markdown
## レビュー必須サイクル

- すべての成果物は別エージェントから skills レビュー最低 1 回（実装→`/review-changes` 等）
- 合格まで反復、3 回失敗時は `issues/open/` に起票して当該タスク中断
- レビュアには git diff と評価基準のみ渡す（実装意図・会話履歴は渡さない）
- 詳細: [agents.md](agents.md)
```

### D. プロジェクトスキル / スラッシュコマンド管理

| 配置先 | 共有範囲 |
|-------|---------|
| `.claude/commands/*.md` | リポジトリ単位（git 共有） |
| `~/.claude/commands/*.md` | 個人ユーザー全体 |
| `.claude/skills/*/SKILL.md` | **時々しか使わない知識** の最適配置（公式推奨） |
| `.claude/agents/*.md` | 専門サブエージェント定義 |

設計原則: 自己完結（他リポジトリのパスをハードコードしない）／詳細基準は `docs/` に分離／CLAUDE.md には一覧と decision tree のみ載せる。

### E. ガバナンス（drift 防止）

CLAUDE.md は時間経過で必ず drift する。「**新しい X を追加する手順**」を CLAUDE.md 内に明記しておくと、人間/AI 後続作業者が同じ構造を維持できる。

例:
```markdown
## 新しいスキルを追加する
1. `.claude/commands/[name].md` を作成（自己完結）
2. 詳細基準は `docs/[topic].md`
3. CLAUDE.md / README.md のテーブル追記
4. エージェント分離が必要なら agents.md にも追記
```

### F. 意思決定支援パターン

CLAUDE.md には**「何を使えばいいか」の判断材料**を含める:

1. **Decision tree**（疑似ツリー）
   ```
   何をレビューしたいか？
   ├─ 実装コード全体    → /review-implementation
   ├─ 直近の変更差分    → /review-changes
   └─ ドキュメント類    → docs/review-*.md
   ```
2. **前提条件テーブル**（「○○があると精度が上がる／なければ××で代替」）
3. **自検証宣言**（条文番号方式: 「第○条に従い…」と Claude に宣言させて遵守ループを作る）

---

### Step 5 完了後の追加ゲート: 別エージェントによる skills レビュー

自己評価が全 Y でも、**それだけでは出力しない**。生成した CLAUDE.md 自身を別エージェントに渡し `docs/review-skill.md` または `docs/review-implementation.md` 相当の skill でレビューさせる:

1. レビュアエージェントに渡すもの: 生成 CLAUDE.md + 公式 Best Practices URL + `docs/review-skill.md`
2. 渡さないもの: 本プロンプト・生成過程の試行錯誤・ユーザーとの会話履歴
3. 合格基準: 90 点以上
4. 不合格なら指摘に従って修正→再レビュー
5. **3 回失敗したら `issues/open/[ID]-claude-md-generation.md` に起票して中断、ユーザーに報告**

## 設計選択: 条文番号方式 vs シンプル方式

| プロジェクト規模 | 推奨フォーマット |
|---------------|---------------|
| 小〜中規模・短期 | シンプル（テンプレートそのまま） |
| 大規模・長期運用・複数 AI 並行 | **条文番号方式**: ルールに第○条と番号を振り、「第○条に従い…」と Claude に宣言させて自検証ループを作る。本文は `.claude/rules/*.md` で管理 |

条文方式は運用コストが高い。**規模に見合った形式を選ぶ**。

## 含めてはいけないもの（再掲・違反したら破棄）

- 一般的な開発常識（「テストを書きましょう」「セキュリティに注意」）
