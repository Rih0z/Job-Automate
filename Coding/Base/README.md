# AI活用型ソフトウェア開発プロセス管理システム

## 概要

このシステムは、AI（Claude等のLLM）を活用してソフトウェア開発の各フェーズを効率的かつ高品質に進めるための**3 Agentシステム**です。

3つの専門的なAIエージェントが協働することで、プロジェクト管理、実装、品質保証を高度に統合し、一貫性のある開発プロセスを実現します。

### 🎯 3つのエージェント（必ず別々のセッションで実行）

1. **リーダーエージェント** - プロジェクト全体を統括・調整（ターミナル1）
2. **実行エージェント** - 設計・実装を担当（ターミナル2）
3. **レビューエージェント** - 品質保証を担当（ターミナル3）

### 🔑 最重要：セッションの完全分離

**このシステムの成功の鍵は、3つのエージェントを必ず別々のAIセッション（別々のターミナル）で実行することです。**

- リーダーエージェント = ターミナル1
- 実行エージェント = ターミナル2  
- レビューエージェント = ターミナル3

同じセッションで複数の役割を実行すると、客観性が失われ、システムが正しく機能しません。ClaudeCode、Gemini-Cli、ChatGPTなど、どのツールを使っても構いませんが、**必ず3つの独立したセッション**で実行してください。

## 3つのエージェントの役割

### 🎩 リーダーエージェント（統括・調整）

プロジェクト全体を俯瞰し、開発プロセスを管理します。

**主な責務**:
- プロジェクトの全体像把握と方向性の決定
- 各フェーズの進行管理とスケジュール調整
- 実行エージェントとレビューエージェントへの指示
- フェーズ間の整合性確保
- 品質基準の設定と判断
- ステークホルダーとのコミュニケーション

**こんな時に活用**:
- プロジェクト開始時の全体計画立案
- 各フェーズ開始前の準備と指示出し
- レビュー結果を受けての判断（次へ進むか、改善するか）
- プロジェクト全体の進捗確認
- 問題発生時の対応方針決定

### 👨‍💻 実行エージェント（設計・実装）

各フェーズの具体的な作業を実行します。

**主な責務**:
- 要件定義書の作成
- システム設計（基本設計・詳細設計）
- コードの実装
- テストケースの作成と実行
- 技術的な判断と実装方針の決定

**こんな時に活用**:
- リーダーからの指示に基づく各フェーズの作業
- 設計ドキュメントの作成
- 実際のコーディング
- レビュー指摘事項の修正

### 🔍 レビューエージェント（品質保証）

成果物を客観的に評価し、品質を保証します。

**主な責務**:
- 成果物の批判的レビュー
- 100点満点での定量的評価
- 改善提案の具体的な提示
- 品質基準への適合性チェック
- リスクや問題点の指摘

**こんな時に活用**:
- 各フェーズ完了後の成果物レビュー
- コード品質のチェック
- 設計の妥当性検証
- セキュリティや保守性の評価

## 対応する開発フェーズ

このシステムは、ソフトウェア開発の以下の5つのフェーズに対応しています：

1. **要件定義** - システムの目的と機能を明確化
2. **基本設計** - システム全体のアーキテクチャを設計
3. **詳細設計** - 各コンポーネントの詳細仕様を作成
4. **実装** - 実際のコードを記述
5. **テスト** - 品質を検証

各フェーズでリーダーが統括し、実行エージェントとレビューエージェントが協働することで、高品質な成果物を段階的に作成できます。

## ⚠️ 最重要：3つのセッションの完全分離

**リーダーエージェント、実行エージェント、レビューエージェントは、必ず3つの別々のAIセッション（3つの別々のターミナル）で実行してください。**

### なぜ3つのセッションを分ける必要があるのか

同じAIセッションで複数の役割を実行すると、以下の重大な問題が発生します：

- **❌ 客観性の完全な喪失**: 実行エージェントが自分の作成した成果物を自分でレビューすることになる
- **❌ バイアスの発生**: 作成時の意図や文脈、前提条件に引きずられる
- **❌ 役割の混同**: リーダー、実行、レビューの境界が曖昧になり、専門性が発揮できない
- **❌ 品質保証の破綻**: 第三者視点での厳格なチェックができない

**3つの独立したセッション**で実行することで：

- ✅ **明確な役割分担**: 各エージェントが自分の専門分野に集中できる
- ✅ **完全に独立したレビュー**: レビューエージェントは成果物のみを見て、バイアスなく評価
- ✅ **客観的な統括**: リーダーエージェントは全体を俯瞰し、公平な判断ができる
- ✅ **厳格な品質管理**: 作成者とは完全に独立した第三者による品質保証

### 推奨される実行方法

**必須条件：必ず3つの独立したターミナル・セッションを使用**

各エージェントは必ず別々のターミナルで別々のセッションとして起動してください。

#### パターン1: ClaudeCode のみを使用（3セッション）

```bash
# ターミナル1（リーダーエージェント専用）
claudecode

# ターミナル2（実行エージェント専用）
claudecode

# ターミナル3（レビューエージェント専用）
claudecode
```

**重要**: 3つの独立したClaudeCodeセッションを**必ず3つの別々のターミナル**で起動します。

#### パターン2: 異なるAIツールを組み合わせ

```bash
# ターミナル1（リーダーエージェント専用）- より高度な判断力
claudecode  # または ChatGPT、Claude Pro など

# ターミナル2（実行エージェント専用）- コーディング特化
claudecode

# ターミナル3（レビューエージェント専用）- 批判的思考
gemini-cli  # または別のClaudeCode セッション
```

異なるAIツールを使用することで、より多様な視点での開発が可能です。

#### パターン3: その他の組み合わせ

以下のような組み合わせも可能です：
- Claude Pro（リーダー） + ClaudeCode（実行） + Gemini-Cli（レビュー）
- ClaudeCode（リーダー） + ClaudeCode（実行） + ChatGPT（レビュー）
- Gemini-Cli（リーダー） + ClaudeCode（実行） + ClaudeCode（レビュー）

**最重要事項：どのツールを使用する場合でも、必ず3つの別々のターミナルで3つの別々のセッションを起動してください。**

### 3 Agentの実行フロー

```
【ターミナル1: リーダー】プロジェクト開始、フェーズ計画
    ↓ 指示書を作成
    
【ターミナル2: 実行】指示書に基づき実行プロンプトを実行
    ↓ 成果物が .tmp/ に保存
    
【ターミナル3: レビュー】成果物を読み込み、レビュープロンプトを実行
    ↓ レビュー結果が .tmp/ に保存
    
【ターミナル1: リーダー】レビュー結果を確認し、判断
    ├─ 70点以上 → 次フェーズへ進行指示
    └─ 70点未満 → 改善指示を実行エージェントへ
         ↓
    【ターミナル2: 実行】改善版を作成
         ↓
    【ターミナル3: レビュー】再レビュー
    （改善サイクルを繰り返す）
```

### ❌ 絶対にやってはいけないこと

```bash
# 同じセッションで複数の役割を実行 - これは絶対にNG！
claudecode
> [リーダープロンプトを実行]
> [同じセッションで実行プロンプトを実行]    # ← 役割が混同、客観性喪失
> [同じセッションでレビュープロンプトを実行]  # ← システムが機能しない
```

```bash
# 2つのセッションだけで実行 - これもNG！
# ターミナル1
claudecode
> [リーダープロンプトを実行]
> [同じセッションで実行プロンプトを実行]  # ← リーダーと実行が混同

# ターミナル2
claudecode
> [レビュープロンプトを実行]
```

**正しい方法：必ず3つの独立したセッション**
```bash
# ターミナル1 - リーダーエージェント専用
claudecode
> [リーダープロンプトのみを実行]

# ターミナル2 - 実行エージェント専用  
claudecode
> [実行プロンプトのみを実行]

# ターミナル3 - レビューエージェント専用
claudecode
> [レビュープロンプトのみを実行]
```

## 基本的な使い方

### 📥 プロンプトの入手方法

以下のいずれかの方法でプロンプトを入手してください：

1. **GitHub から直接コピー（推奨）**
   - [リーダープロンプト](https://github.com/Rih0z/Job-Automate/blob/main/Coding/Base/%E3%83%AA%E3%83%BC%E3%83%80%E3%83%BC%E3%83%97%E3%83%AD%E3%83%B3%E3%83%97%E3%83%88.md)をブラウザで開く
   - 「Raw」ボタンをクリックして、内容をコピー
   - 同様に[実行プロンプト](https://github.com/Rih0z/Job-Automate/blob/main/Coding/Base/%E5%AE%9F%E8%A1%8C%E3%83%97%E3%83%AD%E3%83%B3%E3%83%97%E3%83%88.md)を取得
   - 同様に[レビュープロンプト](https://github.com/Rih0z/Job-Automate/blob/main/Coding/Base/%E3%83%AC%E3%83%93%E3%83%A5%E3%83%BC%E3%83%97%E3%83%AD%E3%83%B3%E3%83%97%E3%83%88%20%C2%B7%20MD)を取得

2. **ローカルファイルから使用**
   - ダウンロードした3つのプロンプトファイル（`リーダープロンプト.md`、`実行プロンプト.md`、`レビュープロンプト.md`）を使用

### ステップ1: リーダーエージェント - プロジェクト開始

1. **新しいターミナル1を開き**、リーダープロンプトを使用
2. [リーダープロンプト](https://github.com/Rih0z/Job-Automate/blob/main/Coding/Base/%E3%83%AA%E3%83%BC%E3%83%80%E3%83%BC%E3%83%97%E3%83%AD%E3%83%B3%E3%83%97%E3%83%88.md)に以下を入力：
   - **プロジェクトの概要**: 作りたいものの説明
   - **現在の状況**: 新規プロジェクトか、途中参加か
   - **特別な要件や制約**: あれば記載
3. リーダーAIがプロジェクト全体計画と最初のフェーズの指示書を作成
4. 指示書が `.tmp/leader_instructions/` 配下に保存される

### ステップ2: 実行エージェント - 作業実行

1. **新しいターミナル2を開き**、実行プロンプトを使用
2. [実行プロンプト](https://github.com/Rih0z/Job-Automate/blob/main/Coding/Base/%E5%AE%9F%E8%A1%8C%E3%83%97%E3%83%AD%E3%83%B3%E3%83%97%E3%83%88.md)に以下を入力：
   - **現在のフェーズ**: リーダーが指示したフェーズ
   - **リーダーからの指示**: `.tmp/leader_instructions/` の指示書
   - **前フェーズの成果物**: 2フェーズ目以降の場合
3. 実行AIが作業を行い、成果物を生成
4. 成果物が `.tmp/[フェーズ名]_[日時]/` 配下に保存

### ステップ3: レビューエージェント - 品質確認

1. **新しいターミナル3を開き**、レビュープロンプトを使用
2. [レビュープロンプト](https://github.com/Rih0z/Job-Automate/blob/main/Coding/Base/%E3%83%AC%E3%83%93%E3%83%A5%E3%83%BC%E3%83%97%E3%83%AD%E3%83%B3%E3%83%97%E3%83%88%20%C2%B7%20MD)に以下を入力：
   - **レビュー対象フェーズ**: 実行エージェントが作業したフェーズ
   - **成果物ドキュメント**: `.tmp/` 配下の `document.md`
   - **メタデータ**: `.tmp/` 配下の `metadata.json`
3. レビューAIが100点満点で評価し、詳細なレビューレポートを作成
4. レビューレポートが `.tmp/[フェーズ名]_[日時]_review/` に保存

### ステップ4: リーダーエージェント - 判断と次アクション

1. **ターミナル1のリーダー**に戻る
2. レビュー結果を確認し、判断：
   - **70点以上**: 次フェーズへ進行する指示書を作成
   - **70点未満**: 改善指示書を作成
3. 指示書に基づき、ステップ2（実行）へ戻る

### ステップ5: サイクルの繰り返し

各フェーズで上記のステップ2〜4を繰り返し、すべてのフェーズを完了させます。

## 詳細な使用例

### 例1: 新規プロジェクトを3 Agentで開始する場合

#### Phase 0: リーダーによる初期計画

**【ターミナル1: リーダー】リーダープロンプトへの入力**:
```
プロジェクト概要:
社内の在庫管理を効率化するWebアプリケーションを作りたい。
主な機能は以下の通り：
- 商品の登録・編集・削除
- 在庫数のリアルタイム表示
- 入出庫履歴の記録
- 在庫が少なくなったら通知
- CSVでのエクスポート

現在の状況: 新規プロジェクト

特別な要件:
- 中小企業向けなので、シンプルで使いやすいUI
- 将来的にモバイルアプリも検討
- セキュリティは標準的なレベルで十分
```

**リーダーの出力**:
- プロジェクト全体計画書
- 要件定義フェーズの指示書（`.tmp/leader_instructions/phase1_requirements.md`）

#### Phase 1: 要件定義フェーズ

**【ターミナル2: 実行】実行プロンプトへの入力**:
```
現在のフェーズ: 要件定義

リーダーからの指示:
[phase1_requirements.md の内容をコピー&ペースト]
```

**実行後**: `.tmp/要件定義_20250130_143000/document.md` が生成される

**【ターミナル3: レビュー】レビュープロンプトへの入力**:
```
レビュー対象フェーズ: 要件定義

成果物ドキュメント:
[document.md の内容をコピー&ペースト]

メタデータ:
[metadata.json の内容をコピー&ペースト]
```

**レビュー結果**: 85/100点、「良好」評価

**【ターミナル1: リーダー】レビュー結果の確認と判断**:
```
レビュー結果を確認しました:
[review_report.md の内容を要約して入力]

85点で良好な評価です。次のフェーズに進みます。
基本設計フェーズの指示書を作成してください。
```

**リーダーの出力**:
- 基本設計フェーズの指示書（`.tmp/leader_instructions/phase2_basic_design.md`）
- 要件定義からの引き継ぎ事項
- レビューで指摘された改善点の反映指示

#### Phase 2: 基本設計フェーズ

**【ターミナル2: 実行】実行プロンプトへの入力**:
```
現在のフェーズ: 基本設計

リーダーからの指示:
[phase2_basic_design.md の内容]

前フェーズの成果物:
[要件定義の document.md の内容]

レビューからの改善事項:
[レビューレポートの「改善提案」部分]
```

このように各フェーズを繰り返していきます。

### 例2: プロジェクト途中からリーダーを投入

既にいくつかのフェーズが完了している場合：

**【ターミナル1: リーダー】リーダープロンプトへの入力**:

```
プロジェクト概要:
[プロジェクトの説明]

現在の状況: 
要件定義と基本設計が完了しており、詳細設計から開始します。

既存の成果物:
- 要件定義書: [document.md の内容]
```


## 評価基準

レビューエージェントは、各フェーズごとに異なる評価項目で100点満点の評価を行います。

### 評価ランク

- **90-100点**: 優秀 - そのまま次フェーズへ進める高品質な成果物
- **70-89点**: 良好 - 軽微な改善推奨事項はあるが、次フェーズに進んでOK
- **50-69点**: 改善必要 - 重要な改善事項があり、修正後に再レビュー推奨
- **0-49点**: 不合格 - 重大な問題があり、大幅な見直しが必要

### リーダーの判断基準

リーダーエージェントは、レビュー結果を基に以下の判断を行います：

- **70点以上**: 次のフェーズへ進行指示
- **70点未満**: 改善指示を出し、改善サイクルへ
- **50点未満**: 根本的な見直しが必要と判断し、フェーズの再設計を指示

## 3 Agentシステムの利点

### 🎯 明確な役割分担

各エージェントが専門分野に集中できるため、高品質な成果物が得られます：

- **リーダー**: 全体最適、判断、調整に専念
- **実行**: 技術的な実装に専念
- **レビュー**: 品質保証に専念

### 🔄 客観的なプロセス

実行者とレビュー者が完全に分離されることで：

- バイアスのない評価
- 厳格な品質基準の適用
- 建設的なフィードバック

### 📊 統合的なプロジェクト管理

リーダーエージェントが全体を統括することで：

- 一貫した方向性の維持
- フェーズ間の整合性確保
- 迅速な問題解決と方針転換

### ⚡ 効率的な改善サイクル

3つのエージェントの協働により：

- 問題の早期発見
- 的確な改善指示
- スムーズなフィードバックループ

## 効果的な使い方のコツ

### 1. リーダーを最大限活用する

リーダーエージェントには、プロジェクトの背景、制約、優先事項を詳しく伝えましょう。リーダーの判断の質は、提供される情報の質に依存します。

### 2. 段階的に進める

一度にすべてを完璧にしようとせず、各フェーズを確実に完了させながら進めましょう。

### 3. レビューを真摯に受け止める

レビューエージェントは批判的ですが、これは品質向上のためです。フィードバックを活かして改善しましょう。

### 4. エージェント間のコミュニケーションを重視

各エージェントに前のエージェントの出力を正確に伝えることで、より良い連携が実現します。

### 5. ドキュメントを蓄積する

各フェーズのドキュメントは `.tmp/` 配下に保存されます。プロジェクトの資産として大切に保管しましょう。

### 6. 反復を恐れない

一度で100点を取る必要はありません。レビュー→改善→再レビューのサイクルを回すことで品質が向上します。

### 7. リーダーの判断を信頼する

リーダーエージェントはプロジェクト全体を見ています。その判断を尊重し、従うことで、全体最適な開発が実現します。

## トラブルシューティング

### Q: 3つのセッションを管理するのが大変

**A**: 各ターミナルに明確な名前をつける（例: 「リーダー」「実行」「レビュー」）か、tmuxやscreenなどのターミナルマルチプレクサを使用すると管理しやすくなります。

### Q: リーダーの指示が曖昧

**A**: リーダープロンプトに、より詳細なプロジェクト情報や制約条件を入力してください。リーダーは提供された情報に基づいて判断します。

### Q: 実行とレビューの意見が対立する

**A**: リーダーエージェントに両方の意見を伝え、判断を仰いでください。リーダーが最終判断を下します。

### Q: AIが出力を途中で止めてしまう

**A**: プロンプトが長すぎる可能性があります。前フェーズのドキュメントを要約して入力するか、重要な部分のみを抜粋してください。

### Q: 評価が厳しすぎる

**A**: レビューエージェントは意図的に厳格です。70点以上あれば十分な品質です。完璧を求めすぎず、実用的なレベルを目指しましょう。

### Q: 専門用語が多くて理解できない

**A**: リーダープロンプトに「専門用語を平易な言葉で説明するよう、実行エージェントに指示してください」と追記することで、よりわかりやすい成果物が得られます。

### Q: .tmpフォルダが見つからない

**A**: リーダーエージェントに「.tmpフォルダを作成してください」と依頼するか、手動でプロジェクトルートに作成してください。

### Q: どのAIツールをどの役割に使うべき？

**A**: 一般的な推奨：
- **リーダー**: より高度な推論能力を持つモデル（Claude Sonnet 4、GPT-4など）
- **実行**: コーディングに強いモデル（ClaudeCode、GitHub Copilot等）
- **レビュー**: 批判的思考に優れたモデル（どのモデルでも可）

ただし、最も重要なのは「3つの別々のセッション」であることです。

### Q: 3つのエージェントを同じセッションで実行してしまった

**A**: システムが正しく機能していません。必ず以下を守ってください：
- リーダーエージェント = ターミナル1の専用セッション
- 実行エージェント = ターミナル2の専用セッション
- レビューエージェント = ターミナル3の専用セッション

同じセッションで複数の役割を実行すると、客観性が完全に失われ、品質保証が機能しません。3つの新しいターミナルを開き、それぞれ独立したセッションで最初から実行し直してください。

### Q: 2つのセッションだけで実行してしまった

**A**: 3つのエージェントすべてに独立したセッションが必要です。たとえリーダーと実行を分けても、レビューが同じセッションだと客観的な評価ができません。必ず3つの独立したターミナル・セッションを使用してください。

## カスタマイズ

このプロンプトシステムは、あなたのプロジェクトに合わせてカスタマイズできます：

### リーダーの判断基準調整

プロジェクトの性質に応じて、リーダーに以下を指示できます：
- 品質重視（90点以上で次フェーズ）
- スピード重視（60点以上で次フェーズ）
- バランス型（70点以上、デフォルト）

### 評価項目の調整

レビューエージェントの評価項目や配点は、プロジェクトの重要事項に合わせて変更できます。

例：セキュリティ重視のプロジェクトでは、セキュリティ項目の配点を高くする

### フェーズの追加・省略

プロジェクトの規模に応じて、フェーズを追加・省略できます。

- 小規模: 要件定義 → 実装 → テスト
- 大規模: すべてのフェーズを使用
- 追加: 運用設計、移行計画などのフェーズを追加

### 成果物フォーマットの変更

document.mdの構造は、組織の標準に合わせて変更できます。

## ベストプラクティス

1. **【最重要】3つのセッションの完全分離を厳守**: リーダー、実行、レビューは必ず3つの別々のターミナル・別々のセッションで実行
   - リーダーエージェント = ターミナル1専用
   - 実行エージェント = ターミナル2専用
   - レビューエージェント = ターミナル3専用
   - この分離がなければシステムは正しく機能しません
2. **リーダーへの詳細な情報提供**: プロジェクトの背景、制約、優先事項を詳しく伝える
3. **定期的なレビュー**: 各フェーズ完了時に必ずレビューを実施
4. **リーダーの判断を尊重**: 最終判断はリーダーエージェントに委ねる
5. **ドキュメント管理**: .tmpフォルダを定期的にバージョン管理システムにコミット
6. **チームでの共有**: レビュー結果やリーダーの判断をチームで共有し、知見を蓄積
7. **改善の記録**: よく出る改善提案はチェックリストとして標準化
8. **柔軟な運用**: プロジェクトの状況に応じて、プロンプトを適宜調整
9. **エージェント間の連携**: 各エージェントの出力を次のエージェントに正確に伝達
