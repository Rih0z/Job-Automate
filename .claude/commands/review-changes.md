# /review-changes — 変更レビュー（エージェント分離実行）

> **このコマンドは別エージェントを起動してレビューを行う。現在のセッションでは直接レビューしない。**
> 実装コンテキスト（思考過程・設計判断・会話履歴）をレビューエージェントに渡さないことで客観性を確保する。

---

## 実行手順

### ステップ1: 情報収集（この実装セッションで行う）

以下の事実情報のみを収集する。実装の意図・目的・設計判断の説明は**含めない**。

```
収集対象:
1. git diff HEAD~1 (直近コミットの差分)
2. git diff --cached (ステージ済み変更)
3. git diff (未ステージ変更)
4. git log --oneline -5 (直近5コミット)
5. 変更されたファイルの一覧と内容
6. テスト設定ファイルの検出と内容:
   - package.json / pyproject.toml / Cargo.toml / go.mod 等
   - jest.config.* / vitest.config.* / pytest.ini 等
   - .github/workflows/ 等のCI設定
7. docs/review-changes.md の内容（評価基準 — プロジェクト内にあれば読み込む）
```

**スコープの優先順位:**
- ステージ済み変更がある場合 → ステージ済み + 直近コミットを主スコープとする
- ステージ済み変更がない場合 → 直近コミットの差分を主スコープとする
- 未ステージ変更は「参考」として記載するが主スコープには含めない

引数が指定された場合:
- ファイルパスの場合 → そのファイルの変更に絞る
- コミット範囲（例: `HEAD~3..HEAD`）の場合 → その範囲の差分を使う

### ステップ2: レビューエージェント起動

**Agent ツール**を使い、以下の設定で別エージェントを起動する:

```
Agent ツールの設定:
- subagent_type: "general-purpose"
- description: "Review changes objectively"
- prompt: 以下のテンプレートに収集した情報を埋め込む
```

**プロンプトテンプレート（レビューエージェントに渡す内容）:**

```
あなたは実装者とは別の客観的なシニアQAエンジニアです。
以下の変更に対し、実装の正確性とテストの網羅性を100点満点で評価してください。

## 行動原則
- Devil's Advocate: 「動いているから正しい」ではなく「本当に正しいか」を疑う
- 証拠主義: 推測でスコアを付けない。コードの事実のみで評価する。証拠不十分時は各レンジの下限で採点する
- 各軸のスコアには、該当するコード箇所（ファイルパスと行番号）を最低1つ以上引用すること
- 非破壊: レビューのみ行い、コードの編集・実行は行わない
- 本スキルはコード実行を行わないため、テストのパス/フェイルは静的分析のみで判断する。CI結果が利用可能な場合はその情報も参照する
```

### ステップ3: 結果の報告

レビューエージェントから返却された結果を、そのままユーザーに表示する。
要約や解釈を加えない（レビューの客観性を維持するため）。

---

## 注意事項

- **絶対に実装セッション内で直接レビューしない**。必ず Agent ツールで別エージェントを起動すること
- レビューエージェントに渡す情報は事実（コード差分・テスト設定）のみ。実装の意図・設計判断は渡さない
- 変更規模が小さい場合（10行以下）でも分離実行する。規模で判断しない
- 詳細なアーキテクチャは `agents.md` を参照
