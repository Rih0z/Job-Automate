# エージェント分離アーキテクチャ

> 実装とレビューを**別エージェント（別セッション）**で実行し、客観的な品質保証を行うための設計。

---

## なぜエージェントを分離するか

同一セッションでの実装とレビューには以下の問題がある：

1. **コンテキスト汚染**: 実装時の思考過程・設計判断を知っているため、「なぜそうしたか」を好意的に解釈してしまう
2. **確証バイアス**: 自分が書いた（または書く過程を見た）コードの問題を見つけにくい
3. **甘い採点**: 実装の苦労を知っているため、無意識にスコアを上げてしまう

**解決策**: レビューエージェントには **コードと差分のみ** を渡し、実装の意図・設計判断・会話履歴は渡さない。

---

## エージェント一覧

| エージェント | 役割 | 起動方法 | コンテキスト分離 |
|-------------|------|----------|----------------|
| **review-changes** | 変更差分のテスト・実装レビュー | `/review-changes` → Agent ツールで自動起動 | 実装コンテキストを遮断。コード差分のみ渡す |

## review-changes エージェント

### 概要
直近の変更に対して、実装の正確性とテストの網羅性を客観的に100点満点で評価する。

### 分離ルール

```
実装セッション（あなた）           レビューエージェント（別プロセス）
┌─────────────────────┐          ┌──────────────────────────┐
│ 実装の思考過程       │    ✗     │                          │
│ 設計判断の理由       │ ───×───→ │  受け取るもの:            │
│ ユーザーとの会話     │          │  - git diff（コード差分） │
│ 試行錯誤の履歴       │          │  - 変更ファイル一覧       │
│                     │    ✓     │  - テスト設定ファイル     │
│ git diff            │ ────────→│  - docs/review-changes.md │
│ 変更ファイル一覧     │          │    （評価基準）           │
│ テスト設定           │          │                          │
└─────────────────────┘          └──────────────────────────┘
```

**渡してよいもの（事実情報のみ）:**
- `git diff` / `git log` の出力
- 変更されたファイルの内容
- テスト設定ファイル（jest.config.*, pytest.ini 等）
- CI設定ファイル
- `docs/review-changes.md` の評価基準

**渡してはいけないもの（実装コンテキスト）:**
- 実装の意図・目的の説明
- 設計判断の理由
- 「ここは後で直す予定」等の予告
- ユーザーとの会話履歴
- 実装セッションでの試行錯誤

### 起動手順（自動）

`/review-changes` コマンドを実行すると、以下が自動で行われる：

1. **情報収集**（実装セッション側）
   - `git diff HEAD~1` / `git diff --cached` / `git diff` を取得
   - 変更ファイル一覧を取得
   - テスト設定ファイルを検出
   - `docs/review-changes.md` の内容を取得

2. **レビューエージェント起動**（Agent ツール）
   - 収集した情報のみをプロンプトに含める
   - `docs/review-changes.md` の評価基準をプロンプトに含める
   - Agent ツールの `subagent_type: "general-purpose"` で起動

3. **結果の返却**
   - レビューエージェントの評価結果を実装セッションに返す
   - ユーザーに構造化されたレビューレポートを表示する

### 手動での分離実行

自動起動ではなく手動で分離したい場合：

```
ターミナル1（実装セッション）:
  → コードを実装する

ターミナル2（レビューセッション — 別の Claude Code を起動）:
  → docs/review-changes.md を読み込ませる
  → 「このプロジェクトの直近の変更をレビューしてください」と依頼
  → レビューエージェントは git diff を自分で取得してレビューする
```

## 新しいエージェントを追加する方法

1. このファイルの「エージェント一覧」テーブルに追記する
2. `/[コマンド名]` のコマンドファイルを `~/.claude/commands/` に作成する（オーケストレーター）
3. 評価基準を `docs/review-[対象].md` に作成する
4. 分離ルール（何を渡すか・何を渡さないか）を明記する
5. `CLAUDE.md` のスキルテーブルに追記する

### エージェント設計の原則

| 原則 | 説明 |
|------|------|
| **コンテキスト最小化** | エージェントに渡す情報は必要最小限の事実のみ |
| **証拠主義** | 推測でスコアを付けない。コードから確認できる事実のみで評価 |
| **非破壊** | レビューエージェントはコードを編集しない。レビューのみ |
| **再現性（ベストエフォート）** | 同じ入力に対して近い評価が得られること。LLMベースのため完全な再現性は保証されない |
| **採点の厳格性** | 証拠不十分時は各レンジの下限で採点。好意的解釈をしない |

---

## 制約と限界

| 制約 | 説明 | 緩和策 |
|------|------|--------|
| **コンテキスト遮断はベストエフォート** | Agent ツールは別プロセスだが、完全な情報遮断を技術的に保証するものではない | 手動分離（別ターミナル）を併用することで確実性を上げる |
| **静的分析のみ** | テストの実行結果（パス/フェイル）は確認できない | CI結果やテスト実行ログがあれば併せて渡す |
| **LLMの非決定性** | 同じ入力でも毎回微妙に異なる評価になりうる | 複数回実行して傾向を確認する。スコアの±5点は誤差範囲と捉える |
| **プロンプト依存** | レビュー品質はプロンプトの指示に依存する | 定期的にレビュー結果を人間が検証し、プロンプトを改善する |
