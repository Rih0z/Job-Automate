# OneNEC HTML記事 品質レビュープロンプト

> **対象**: `make-html.md` プロンプトをもとに作成された OneNEC 社内ブログ用 HTML 記事
> **目的**: 「情報の正確性」「読者ペルソナへの適合性」「事実と考察の分離」「客観的評価の可能性」の4軸＋「OneNEC技術準拠」の観点から記事品質を100点満点で評価する。

---

## レビュー前の必須確認事項

レビューを開始する前に、以下を記事または執筆者から確認すること。

| 確認項目 | 確認内容 |
|----------|----------|
| **この記事で最も伝えたいことは何か？** | 主張・結論を1〜2文で言語化できるか |
| **想定読者は誰か？** | OneNEC読者（NEC社員全般、新卒含む）のどの層を主対象にしているか |
| **読者がこの記事を読んだ後に何を「できる」「わかる」ようになるか？** | 行動変容または理解のゴールが明示されているか |
| **make-html.md のどのルール群を適用したか？** | 特殊な省略・変更があれば事前に把握する |

上記が不明瞭な場合は、レビュー開始前に執筆者へ確認すること。

## レビュー観点（5項目・各20点）

---

### 1. OneNEC HTML技術準拠・情報正確性 【配点：20点】

> **目的：** `make-html.md` の変換ルールが正しく適用されており、かつ記事に含まれる情報が正確であるかを確認する。

#### 1-A. 必須セクション完備チェック

- [ ] **3分で読める要約**が目次の前に配置されている
- [ ] **関連記事への導入ブロック**が冒頭にあり、以下2記事が常に含まれている
  - `【RTC通信】RAGは本当に必要か？` (post/2635646)
  - `【RTC通信】指示するだけでAIが全部やる時代` (post/2634586)
- [ ] **免責事項**セクションが記事本文末尾（AIモデル切り替えセクションの前）にある
- [ ] **AIモデル切り替えサーバアプリ開発 - 賛同者募集中**セクションがある
- [ ] **AI活動の出資者・協力者を募集しています**ボックスがある
- [ ] **他の記事はこちら**セクションがある（記事内容に応じた関連記事が選択されている）
- [ ] **組織紹介**セクションがある
- [ ] **フッター**（© NEC Corporation of America）がある
- [ ] **著者**が「RTC Koki Riho（https://github.com/Rih0z）」に設定されている
- [ ] **タイトルと公開日**が削除されている
- [ ] **絵文字**がすべて排除されている

#### 1-B. OneNEC専用フォーマット適用チェック

- [ ] 表が `<div class="table-layer">` でラップされ、`table-headling-x` または `table-headling-y` クラスが適切に選択されている
- [ ] コードブロックに `language-xxxx` クラスが付与されている（`language-plaintext` による代替も可）
- [ ] Q&A形式が `<ul class="ckeditor5-qa">` で記述されている
- [ ] 空のキャプションに `&nbsp;` が使われている（空の `<caption>` タグは使用していない）
- [ ] `AISever v4` 等の表記が「AIモデル切り替えサーバー」に統一されている

#### 1-C. 情報正確性チェック

- [ ] 数値・統計・技術仕様の記述に誤りがないか
- [ ] 専門用語の解説（括弧内の説明）が正確か（例：RAG = Retrieval-Augmented Generation の説明が正しいか）
- [ ] 引用・参照している外部情報のソースが明示されているか
- [ ] 記載している手順・コード・設定値が実際の動作と一致しているか
- [ ] リンクに記載されているURLが過去記事の内容・タイトルと一致しているか

**採点目安**

| 点数 | 基準 |
|------|------|
| 18〜20 | 全必須セクションが揃い、フォーマットも正確。情報の誤りがない |
| 14〜17 | 必須セクションはあるが、一部フォーマット違反または情報の軽微な誤りがある |
| 10〜13 | 必須セクションの欠落または情報の複数箇所に誤りがある |
| 0〜9  | 必須セクションが複数欠落し、情報の信頼性に重大な問題がある |

---

### 2. 読者像の明確化 【配点：20点】

> **目的：** OneNEC読者（NEC社員、特に新卒社員を含む多様な層）を念頭に、記事が「誰のために書かれているか」が冒頭から一貫して明示されているかを確認する。

- [ ] 3分要約に「この記事の対象者」が暗示または明示されているか
- [ ] タイトル・リード文・見出しが想定読者のリテラシーレベルに合っているか
- [ ] 記事を読んで「自分には関係ない」と感じる読者層をできる限り排除する工夫があるか（「〜を担当している方」「〜を知りたい方」のような呼びかけ）
- [ ] AI未経験者や新卒社員でも記事の目的とゴールを冒頭で把握できるか
- [ ] 技術者向けと非技術者向けの内容が混在する場合、適切にセクションが分かれているか

**採点目安**

| 点数 | 基準 |
|------|------|
| 18〜20 | 読者像が冒頭で明確に示され、全体を通じて一貫している |
| 14〜17 | おおむね明確だが、一部で読者像が揺れている |
| 10〜13 | 読者像の定義が曖昧で、誰向けか判断しにくい |
| 0〜9  | 読者像の記述がなく、誰向けかまったく不明 |

---

### 3. 読者ペルソナへの適合性 【配点：20点】

> **目的：** `make-html.md` の「新卒でも理解できるをテーマ（ルール#7・#8）」に基づき、記事の語彙・説明粒度・事例がOneNEC読者ペルソナに最適化されているかを確認する。

- [ ] **専門用語の全てに括弧書き解説が付いているか**（make-html.md ルール#7 準拠）
  - 例：RAG → `RAG（Retrieval-Augmented Generation：情報検索を組み合わせた生成AI技術）`
  - 例：API → `API（Application Programming Interface：ソフトウェア同士が連携するための仕組み）`
- [ ] 解説の前置きが必要な箇所で補足説明が追加されているか（ルール#8）
- [ ] 情報量を減らさずに解説が追加されているか（情報削除による簡略化をしていないか）
- [ ] 事例・比喩・図が、NEC社員の業務環境（社内ツール・OneNEC・Azure等）に即しているか
- [ ] 読者が「自分の業務でどう使えるか」をイメージできるような具体例があるか
- [ ] 暗黙の前提（業界知識・社内用語）に依存した説明になっていないか
- [ ] 説明が過剰に技術的で新卒社員が置いてけぼりになっていないか、あるいは逆に簡略化しすぎて技術者には物足りないと感じないか

**採点目安**

| 点数 | 基準 |
|------|------|
| 18〜20 | 全専門用語に解説があり、情報量を保ちながら新卒にも理解可能な説明 |
| 14〜17 | おおむね適切だが、一部で解説が漏れているか説明粒度がずれている |
| 10〜13 | 専門用語の解説漏れが複数あり、読者の理解に支障をきたす箇所がある |
| 0〜9  | 読者への配慮が不足しており、新卒読者には理解困難な記述が多い |

---

### 4. 事実と考察・意見の分離 【配点：20点】

> **目的：** 客観的事実と著者の解釈・考察・意見が明確に区別されているかを確認する。OneNEC記事は「我々のプロジェクトで得た知見の紹介」という位置付けであるため、特に個人の体験と一般的事実の区別が重要。

- [ ] 事実として断定している記述に、根拠・出典・データが付いているか
- [ ] 著者の意見・推測・解釈には「〜と考えられる」「筆者の見解では」「我々の環境では」などの表現が使われているか
- [ ] RTC（Redmond Technology Center）の環境固有の知見であることが、一般的な情報と混同されずに明記されているか
- [ ] 考察・洞察のセクションが事実提示のセクションと構造的に分かれているか
- [ ] 数値・統計・技術仕様など、検証可能な情報が正確に記載されているか
- [ ] 「事実のように見えるが実は著者の主観または特定環境での結果」となっている記述がないか
- [ ] 免責事項の内容（「NECとは異なる環境」「個人の責任での利用」）が記事本文の記述スタイルと整合しているか

**採点目安**

| 点数 | 基準 |
|------|------|
| 18〜20 | 事実と考察が常に明確に区別されており、RTC環境固有の知見も適切に注記されている |
| 14〜17 | おおむね分離されているが、一部で混在している箇所がある |
| 10〜13 | 事実と意見の混在が複数あり、読者が誤解しやすい |
| 0〜9  | 事実と意見の区別がほとんどなく、信頼性に問題がある |

---

### 5. 合理性と客観的評価の可能性 【配点：20点】

> **目的：** 主張・結論が合理的な根拠に基づき、読者が独立して評価・検証できる記述になっているかを確認する。

- [ ] 主張・推奨・結論に明確な根拠（データ・実験・文献・社内検証結果）が示されているか
- [ ] 反論・代替案・制約条件が公平に取り上げられているか
- [ ] 特定の方向性・結論に誘導するような選択的な事実提示になっていないか
- [ ] 読者が自分で判断・評価できるように、比較軸や評価基準が提供されているか
- [ ] 「〜だからこうすべき」という断定に、それを否定する視点への言及が含まれているか
- [ ] 読者が「自分の環境でも試してみよう」と思えるような再現性への言及（または環境差異への注意喚起）があるか
- [ ] 3分要約に記載された結論が、記事本文の根拠と整合しているか

**採点目安**

| 点数 | 基準 |
|------|------|
| 18〜20 | 根拠・反論・限界が明示され、読者が独立して判断できる |
| 14〜17 | おおむね合理的だが、一部で根拠不足または誘導的な記述がある |
| 10〜13 | 根拠が薄く、特定の方向への誘導が見られる |
| 0〜9  | 主観的主張が多く、客観的評価が困難 |

## 重要な注意事項

> ⚠️ **レビュワーは元ファイルを絶対に編集してはならない**
>
> レビューはあくまで「フィードバックの提供」であり、修正作業は執筆者自身が行う。
> 改善点は具体的に指摘し、修正案は「提案」として記載すること。
>
> また、OneNEC HTMLの技術的な規約（table-layer、ckeditor5-qa等）に起因する問題については、
> make-html.md の該当ルール番号を明記すること。

---

## 使用方法

1. このプロンプトと一緒に、レビュー対象の HTML ファイル（または CKEditor 貼り付け用本文）を添付する
2. 「このOneNEC HTML記事をレビューしてください」と依頼する
3. 「レビュー前の必須確認事項」が文書から読み取れない場合は、執筆者へ確認する
4. 採点結果を `.tmp/` ディレクトリに `review-<記事名>-<YYYYMMDD>.md` 形式で格納する
5. B判定以下の場合は、具体的な修正箇所リストを執筆者に共有し、再レビューを実施する
