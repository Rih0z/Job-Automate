# 汎用ブログ記事レビュープロンプト

## 目的

本プロンプトは、**任意のサイト（個人ブログ、企業ブログ、技術メディア、note、Qiita、Zenn、Medium、Dev.to、社内ポータル等）に公開予定のブログ記事**に対して、品質・有用性・読みやすさ・事実正確性・コンプライアンスの観点から客観的なレビューを実施するためのプロンプトである。

プラットフォーム固有の制約（絵文字エラー等）や組織固有の規約（社内規程等）は**設定欄**でカスタマイズする。

---

## レビュー前の設定欄（必ず記入してから開始）

レビュー開始前に、対象記事に応じて以下を記入する。記入値はレビュー全体に適用される。設定欄は **プラットフォーム軸 × 業界軸 × 記事タイプ軸 × 運用軸** の 4 ブロックで構成。

### A. プラットフォーム軸

| 項目 | 記入例 | 本記事の設定 |
|------|--------|--------------|
| 掲載先プラットフォーム | note / Qiita / Zenn / Medium / WordPress / Hatena / Dev.to / 企業ブログ / 社内ポータル 等 | |
| プラットフォーム既知制約 | 絵文字制限 / Markdown 拡張記法 / Frontmatter 必須 / タグ規約 / 文字数上限 等（0-gate プラットフォーム別早見表参照） | |
| 絵文字ポリシー | 全面禁止 / 装飾系のみ可 / 制限なし / プラットフォーム制約に従う | |

### B. 業界軸

| 項目 | 記入例 | 本記事の設定 |
|------|--------|--------------|
| 業界規制 | 該当なし / 医療（薬機法）/ 金融（金商法）/ 法律（弁護士法72条）/ 旅行（景表法）/ 化粧品 / 食品 / 不動産 / 教育 / 他 | |
| コンプライアンス制約 | 個人情報・NDA・社外秘・著作権 / 業界規制（上記）/ 該当なし | |
| 利益相反開示要否 | 必要（広告記事・自社製品紹介・スポンサー記事）/ 不要 | |

### C. 記事タイプ軸

| 項目 | 記入例 | 本記事の設定 |
|------|--------|--------------|
| 記事タイプ | 概念解説 / ハウツー / ベンチマーク / 製品レビュー / AI 出力評価 / 戦略分析・調査（一次計測なし）/ エッセイ・コラム / 取材・インタビュー / 実証実験 等（早見表参照） | |
| 想定読者層 | 初心者 / 中級エンジニア / 経営層 / 一般ビジネスパーソン / 学生 / シニア層 等 | |
| 記事の目的 | 知見共有 / 集客 / SEO / 採用 / 啓蒙 / 製品紹介 / コミュニティ形成 等 | |
| 外部主体断定の有無 | あり（4-gate 必須実施）/ なし（4-gate skip 可）/ 不明（4-gate 実施推奨） | |

### D. 運用軸

| 項目 | 記入例 | 本記事の設定 |
|------|--------|--------------|
| 文体ポリシー | 敬語（です・ます調）/ 常体（だ・である調）/ カジュアル / 英語 / 指定なし | |
| 目標スコア | 80 点以上（A 判定）等 ※ **100 点換算後の値で指定**（素点ではなく換算点） | |
| 出力フォーマット粒度 | 完全版（11 観点 + 第11項目補助表）/ 最小版（11 観点のみ、第11項目補助表は技術評価記事のみ） | |

## レビュー実施前の確認事項

### 0-gate. プラットフォーム固有制約チェック（必要に応じて発動）

> **重大違反検出時は即D判定**: 掲載先プラットフォームに固有の技術的制約（投稿不能・表示崩れ等の実害が発生するもの）が記事に含まれる場合、他項目スコアに関わらず **判定 D で公開不可** とする。

#### 主要プラットフォーム別 既知制約早見表

| プラットフォーム | 絵文字 | HTML タグ | Frontmatter | 文字数制約 | その他主要制約 |
|---|---|---|---|---|---|
| **note** | 可 | 限定的（script/iframe NG） | 不要 | タイトル 100 字 | 見出し階層浅め、画像 10MB/枚 |
| **Qiita** | 可 | 限定的（独自 Markdown 拡張）| Frontmatter 必須（tags 必須）| なし | タグ 5 個上限、コードブロック言語指定推奨 |
| **Zenn** | 可 | 不可（純 Markdown） | Frontmatter 必須（title/topics/published 等） | なし | chapters 機能（本）、topics 5 個上限 |
| **Medium** | 可 | 限定的 | 不要 | なし | メンバーウォール設定、英文記事も多い、claps 文化 |
| **Hatena Blog** | 可 | 可（HTML 直書き OK） | 不要 | なし | 記法モード（Markdown / はてな記法）選択必須 |
| **Dev.to** | 可 | 限定的 | Frontmatter 必須（cover_image 等） | なし | tags 4 個上限、Liquid タグ対応 |
| **WordPress（自己ホスト）** | 可 | プラグイン依存 | 不要 | なし | テーマ・プラグイン固有制約 |
| **企業エンジニアブログ** | 可（社内ガイドライン依存） | サイト依存 | サイト依存 | サイト依存 | 広報・法務レビュー導線あり、ブランドガイドライン |
| **社内ポータル（CKEditor 系）** | **不可**（投稿エラー） | サニタイズ強め | 不要 | なし | 装飾絵文字（✅❌⚠️等）で CMS システムエラー、NDA 厳格 |

> プラットフォームが上記に無い場合、設定欄記入時に公式ヘルプ・利用規約・投稿ガイドラインを Web 検索で確認すること。

#### 発動例（カテゴリ別）

- **絵文字制限のあるプラットフォーム**: `✅ / ❌ / ✔ / ✘ / 🟢 / 🔴 / ⚠️ / 💡 / 📌` 等の Unicode 絵文字（`U+2600–U+27BF` / `U+1F300–U+1F9FF` / `U+1FA00–U+1FAFF` / `U+2705` / `U+274C` / `U+2714` / `U+2718`）が本文・表・SVG `<text>` 要素・figcaption・alt 属性に含まれる場合、一部 CMS（CKEditor 系等）でシステムエラーが発生し投稿不能となる事例がある。
- **HTML タグ制限**: 一部プラットフォームでは `<script>` / `<iframe>` / 一部 inline 属性が削除される。
- **画像サイズ・形式制限**: WebP 非対応、ファイルサイズ上限、画像枚数上限など。
- **文字数制限**: タイトル文字数、本文最大文字数、見出し階層の深さ制限など。
- **Frontmatter 不整合**: Zenn / Qiita / Dev.to の Frontmatter 必須項目（title / topics / tags / published 等）の欠落・スペルミス・型違反で公開失敗。

#### ゲート通過条件

1. 設定欄「掲載先プラットフォーム」「プラットフォーム既知制約」に応じた制約を**早見表 or 公式ドキュメントで確認**
2. 検出された制約違反をゼロにする
3. 検出された場合は**文字・代替記号・装飾タグに置換**してから再レビュー（例: ✅→「成功」「OK」「可」、❌→「失敗」「NG」「不可」、⚠️→「注意」、視覚強調は `<strong>` / 色付き `<span>` / 背景色付き `<div>` で代替）

#### 絵文字検出コマンド例

```bash
# Markdown / HTML / SVG 等のテキストファイルに装飾絵文字が混入していないか検出
grep -nP '[\x{2600}-\x{27BF}\x{1F300}-\x{1F9FF}\x{1FA00}-\x{1FAFF}\x{2705}\x{274C}\x{2714}\x{2718}]' 対象ファイル

# 出力例（NG）:
# 116:   <strong>✅ 完遂 1,058 s</strong>
# 119:   ❌ stall 失敗
```

PowerShell 環境では:

```powershell
Select-String -Path 対象ファイル -Pattern '[☀-➿\u{1F300}-\u{1F9FF}\u{1FA00}-\u{1FAFF}✅❌✔✘]'
```

**置換後の例（OK）**:

```html
<strong style="color:#2E7D32;">完遂 1,058 s</strong>
<span style="color:#c62828;">stall 失敗</span>
```

**検出ゼロ** であることをゲート通過の必須条件とする。

**過去パターン**: 一部社内 CMS（CKEditor ベース）では `✅ / ❌` を含む HTML を投稿しようとするとシステムエラーで投稿不能になる事例が報告されている。装飾系絵文字の使用可否は**プラットフォーム単位で事前確認**すること。

### 冗長性チェック（レビュー開始前に必須）

記事の内容が冗長・多岐にわたっていると判断した場合、**レビュー実施前**に執筆者へ以下を確認すること。

- **この記事で最も伝えたいことは何ですか？**（1 文で言える核心メッセージ）
- **想定読者は誰ですか？**（設定欄と矛盾していないか）
- **記事のスコープをどこまでに絞りたいですか？**（読了時間の目安）
- **読者に取らせたいアクションは何ですか？**（CTA：購読・ブックマーク・実装・問い合わせ等）

執筆者の意図を確認した上でレビューを実施する。

## レビュー観点（11項目・計110点満点）

### 1. 想定読者向けのわかりやすさ 【配点：10点】

設定欄で指定した「想定読者層」に対する読みやすさを評価する。

- [ ] 想定読者にとって専門用語に適切な説明や補足があるか
- [ ] 想定読者の前提知識を超えない範囲で構成されているか
- [ ] 図表・コード例・具体例で抽象概念が補強されているか
- [ ] 難易度の段階的な説明（既知 → 未知）になっているか
- [ ] 読者が読み始めから読了まで「迷子」にならない導線か

### 2. 独自性・一次情報の含有 【配点：10点】

- [ ] 執筆者自身の経験・検証・調査から得られた独自の知見が含まれているか
- [ ] 一般論の引き写しだけでなく、具体的な事例・エピソード・データがあるか
- [ ] 失敗談・トラブルシューティング・落とし穴の共有があるか
- [ ] 「やってみてわかったこと」「他では見たことがない視点」が明記されているか
- [ ] 既存の解説記事との差別化ポイントが明確か（読者がこの記事を読むべき理由）

### 3. 読者にとっての有益性 【配点：10点】

- [ ] 読了後に読者が具体的なアクションを取れる内容か（実装・購読・調査・議論等）
- [ ] 設定欄で指定した「記事の目的」（知見共有・集客・SEO 等）と整合しているか
- [ ] 読者の課題解決・意思決定・スキル向上に貢献する情報か
- [ ] 想定読者層以外にも応用可能な普遍的な学びが含まれているか

### 4. 事実の正確性と参考情報 【配点：10点】

> **重複圧縮ガイド**: 本項目（観点）↔ 4-gate（公開ブロッカー判定）↔ Phase 4b（公式反論検索エージェント）は **3 階層の責務分担** で構成。観点 = 静的チェック、ゲート = 重大違反判定、Phase = 実行手順。重複読みを避けるため、本項目では**観点のみ**を記載し、検索手順・ゲート判定基準は cross-reference する。

**基本観点（一般事実）**:

- [ ] 記載されている事実は正確か（技術仕様、数値、日付、企業情報、人物名など）
- [ ] 参考文献・リンクが適切に付与されているか
- [ ] 参考リンクは**アクセス可能**で**信頼性のある情報源**か（リンク切れ・低品質ブログのみの典拠を避ける）
- [ ] 公式ドキュメント・一次情報を優先的に参照しているか
- [ ] **イベント・リリース・PR マージ等の日付は年月日まで（重要なら時刻も）明記されているか**（「4 月」ではなく「2026 年 4 月 3 日」のように具体的に）
- [ ] **日付・時刻・数値の整合性が取れているか**（同じ事象について記事内で異なる値が記載されていないか）
- [ ] **数値の単位・換算が正確か**（USD/JPY、tok/s vs ms/token、MB vs MiB、% vs ratio など）※ 単位の **混在チェック** は第7項目「冗長性・一貫性」、**統計的裏付け** は第11項目 Q を参照

**外部主体（企業・組織・個人）に関する断定的主張がある場合**（→ 詳細手順は 4-gate / Phase 4b へ）:

- [ ] **典拠が一次ソース（当事者発言・公式ドキュメント・決算資料・論文）かセカンダリソース（報道・解説記事・ブログ・SNS の伝聞）かを区別しているか**
- [ ] **セカンダリソースのみを典拠とする断定的主張について、当事者の公式反論・否認・訂正声明を Web 検索済みか**（→ Phase 4b 公式反論・否認検索レビューで実施）
- [ ] **当事者の公式反論が見つかった場合、本文で両論併記しているか**
- [ ] **発言者・主張者が特定できる主張は「誰が／いつ／どこで」言ったかを明記しているか**
- [ ] **過去時点の主張と現在時点の状況を混同していないか**（時点明示が必須）

※ 重大違反判定基準 = **4-gate**（本観点直下に記載）／実行手順 = **Phase 4b**（本プロンプト後半「独立エージェントによる多段レビュー」参照）

#### 4-gate. 外部主体の財務・採算性・戦略判断の断定 ゲート条項

> **重大違反検出時は即D判定**: 第4項目で以下のいずれかが検出された場合、**他項目スコアに関わらず判定Dで公開不可** とする（第9-gate と同等）。
>
> - **外部の企業・組織・個人（実在固有名詞）の財務・採算性・戦略判断・性能・評判を "セカンダリソース（報道・解説記事・ブログ・SNS の伝聞）のみ" を根拠に断定** しており、かつ **当事者の公式反論・否認・訂正声明の Web 検索を実施していない**
> - 上記 Web 検索の結果、**公式反論が存在するにも関わらず本文で言及していない**（両論併記になっていない）
> - 過去時点の断定的主張を、時点を明示せず現在時点の文脈で引用している
>
> **発動トリガー（以下キーワードを含む主張が記事にある場合は必ず 4-gate を走らせる）**:
>
> - 財務系: 「赤字」「黒字」「損失」「利益」「採算性」「持続不可能」「ロスリーダー」「Loss Leader」「破産」「資金繰り」
> - 戦略系: 「〜社は〜と判断した」「〜社は〜を狙っている」「〜社の戦略は〜」「〜社は失敗した」「〜社は成功した」
> - 性能・評判系: 「〜は最速」「〜は最低」「〜は使い物にならない」「〜は業界標準」
> - 伝聞系: 「複数の報道によれば」「〜と報じられている」「業界では〜と言われている」「内部関係者によれば」
> - 固有名詞 + 断定: 大企業名・著名人名 + 断定的述語の組合せ
>
> **ゲート通過条件**:
>
> 1. 上記キーワードを含む断定的主張を記事から抽出
> 2. 各主張について「公式反論・否認検索レビュー」（本プロンプト後半参照）を独立エージェントで 1 回実施済み（実施ログを保存）
> 3. 見つかった反論は本文で両論併記、見つからなかった場合は脚注で「セカンダリソース由来、一次ソース未検証」と明記
> 4. 過去時点の主張は時点（例: 「2023 年 10 月の WSJ 報道」）を明示
>
> **過去パターン**: 大手紙の単独報道で「ある製品が発足当初からユーザー1人あたり赤字」と書いた記事に対し、当該製品の初代 CEO が SNS で "False" と公開否認し、別の幹部もカンファレンスで「黒字、$100M ARR」と反論していた事例がある。読者からの指摘で初めて気付くケースが散見されるため、本ゲートで再発を防ぐ。

### 5. 事実と所感の分離 【配点：10点】

- [ ] 客観的事実と執筆者の意見・感想が明確に区別されているか
- [ ] 「〜と思います」「〜と感じました」「〜の可能性がある」など、所感・推測であることが明示されているか
- [ ] 事実として断定している箇所に誤りがないか
- [ ] **「複数の報道によれば」「業界では〜と言われている」等の曖昧な主語で断定していないか**（断定の根拠が不明瞭な主張は、所感扱いとして「可能性」「〜との指摘がある」に書き換えるか、具体的な主体・発言者を明示する）
- [ ] **対立する主張が公知であるトピックについて、片方だけを採用して断定していないか**（両論併記が原則）
- [ ] 主観的体験・感情と、検証可能な事実を段落・記号・見出しで分離しているか

### 6. 冒頭サマリーの有無 【配点：10点】

- [ ] 記事冒頭に要約・サマリーセクションがあるか
- [ ] サマリーを読むだけで記事の概要・結論が把握できるか
- [ ] 「この記事でわかること」「対象読者」「前提知識」が明記されているか
- [ ] 読了時間の目安が記載されているか
- [ ] サマリーが本文の核心と矛盾していないか（タイトル詐欺・サマリー詐欺を防ぐ）

### 7. 冗長性の排除・一貫性 【配点：10点】

- [ ] 本筋から逸れた話題が含まれていないか
- [ ] 同じ内容を繰り返し説明していないか
- [ ] 不要な前置きや脱線がないか
- [ ] 各セクションが記事の目的に沿っているか
- [ ] **記事全体が一貫した文脈で書かれているか**（後から追記した内容が「追記感」を出していないか。最初から一本の記事として書かれたように読めるか）
- [ ] **まとめ・結論が記事全体を総括しているか**（部分的な評価ではなく、全セクションを踏まえた統一的な結論になっているか）
- [ ] **主語・視点が一貫しているか**（「我々が検証した」立場で書いている場合、途中で「読者へのアドバイス」視点に変わっていないか。執筆者・読者・第三者の立場が混在していないか）
- [ ] **複数回のテスト・検証・取材を比較する場合、評価基準・比較単位・環境が統一されているか**（テスト1では「秒」、テスト2では「分」のように単位が混在していないか）
- [ ] **比較表が統一されているか**（同じ列構成・同じ評価項目で並列に並べられているか）
- [ ] **環境・条件の差分が明示されているか**（バージョン、ハードウェア、サンプル、データセット等が変わった場合は変更点を明記）
- [ ] **人が見て一目で比較しやすい構造になっているか**（同じ指標を縦・横で揃え、視覚的に差分が読み取れる表設計）
- [ ] **複数テストの結果が要約・まとめに全て反映されているか**（「3回のテストで毎回改善」と書くなら 3 回分すべての結果がサマリーに掲載されているか）

### 8. 読みやすさ・文体統一 【配点：10点】

- [ ] 適切な見出し・段落分けがされているか
- [ ] 一文が長すぎないか（目安：60〜80 文字以内）
- [ ] 箇条書き・表・コードブロック・図が効果的に使われているか
- [ ] 誤字脱字・表記ゆれがないか（揺れ例: ユーザ/ユーザー、サーバ/サーバー、Web/ウェブ）
- [ ] 論理的な流れで構成されているか（結論先出し or 結論最後を意図的に統一）

|------|------|------|
| S | 99〜110点（90%以上） | 公開可（そのまま掲載OK） |
| A | 88〜98点（80%以上） | 軽微な修正後に公開可 |
| B | 77〜87点（70%以上） | 修正推奨（要フィードバック反映） |
| C | 66〜76点（60%以上） | 大幅な修正が必要 |
| D | 65点以下（59%以下） | 構成から見直しが必要 |

> **ゲート条項による即D判定**: 総合点に関わらず、以下のいずれかが検出された場合は **判定 D で公開不可**:
>
> - **0-gate**（プラットフォーム固有制約違反）: 設定欄の掲載先プラットフォームで投稿不能・表示崩れ等の実害が出る要素
> - **4-gate**（外部主体財務・採算性・性能・評判断定の公式反論未検索）
> - **9-gate**（コンプライアンス・機密情報・著作権・規制違反）

> **N/A処理ルール（粒度：項目全体 / サブセクション / サブ項目の3段階）**
>
> 1. **項目全体N/A**: 第11項目を含まない記事（例: 概念解説のみ、エッセイ、コラム）は、第11項目（10点）を全体N/Aとし、**総合100点満点で採点**する。
> 2. **サブセクション単位N/A**: 第11項目のうち特定サブセクションが適用外の場合（例: 成果物の生成主体帰属 G は AI/LLM 出力を扱わない記事では N/A）、**第11項目の満点をそのサブセクション配点分だけ縮小**する（例: G サブセクション N/A なら第11項目は 9.0 点満点）。
> 3. **サブ項目単位N/A**: サブセクション内の個別チェック項目が適用外の場合、そのチェック項目を減点対象から除外する（配点は変更しない）。
> 4. **100点換算式**: `換算点 = 素点 × 100 / 該当配点合計`（該当配点合計 = 110 − N/A縮小分）。コメント欄に「N/Aで縮小した満点」と「換算前後の点数」を明記する。
>
> **ゲート条項**: 上記ルールに関わらず、4-gate / 9-gate / 0-gate の重大違反が検出された場合は **判定Dで公開不可** とする。

### 採点運用例（参考）

**例1: AI 出力評価記事（110点満点）**

- 第1〜10項目: 各 9.0 点 × 10 = **90.0 点**
- 第11項目:
  - F = 2.0（全満たす）
  - Q = 2.0 − 0.4（Q-4 サンプル数欠落：Nice欠落）= **1.6 点**
  - C = 1.0（全満たす）
  - R = 2.0 − 1.0（R-1 前提条件欠落：Must欠落）= **1.0 点**
  - A = 1.5 − 0.3（A-4 ネガティブ結果欠落：Nice欠落）= **1.2 点**
  - O = 0.5（全満たす）
  - G = 1.0 − 0.5（G-1 生成主体記述曖昧：Must欠落）= **0.5 点**
  - サブ計: 2.0 + 1.6 + 1.0 + 1.0 + 1.2 + 0.5 + 0.5 = **7.8 点**
- 素点: 90.0 + 7.8 = **97.8 点 / 110 点**
- 100 点換算: 97.8 × 100 / 110 = **88.9 点 / 100 点**
- 判定: **A**（88〜98点）

**例2: 概念解説記事（第11項目N/A、100点満点）**

- 第1〜10項目: 各 9.0 点 × 10 = 90 点
- 第11項目: N/A
- 素点: 90 点 / 該当配点合計 100 点
- 100 点換算: **90.0 点 / 100 点**
- 判定: **S**

**例3: エッセイ・コラム（第11項目N/A、100点満点）**

- 第1〜10項目: 各 8.5 点 × 10 = 85.0 点（第5項目「事実と所感の分離」が所感主体記事のため減点目立たず）
- 第11項目: N/A
- 4-gate: 著名人言及あるが伝聞断定なしのため発動なし
- 9-gate: 違反なし
- 素点: 85.0 点 / 100 点
- 100 点換算: **85.0 点 / 100 点**
- 判定: **B**（77〜87点、修正推奨）

**例4: 戦略分析・調査記事（109点満点、F/Q/R 整合性検証モード）**

- 第1〜10項目: 各 9.0 点 × 10 = 90.0 点（4-gate Phase 4b 実施済み、両論併記済みで第4項目満点に近い）
- 第11項目（G を N/A、F/Q/R は整合性検証モード）:
  - F = 2.0（第三者公表値の比較条件の整合性 OK、満点）
  - Q = 2.0 − 0.4（Q-3 測定誤差言及なし、Nice 欠落）= **1.6 点**
  - C = 1.0（評価基準の出典明示 OK）
  - R = 2.0（R-2 第三者バージョン明記、Must 満たし）
  - A = 1.5 − 0.75（A-2 一部論理の飛躍、Must 欠落）= **0.75 点**
  - O = 0.5（公式情報リンク OK）
  - G = N/A
  - サブ計: 2.0 + 1.6 + 1.0 + 2.0 + 0.75 + 0.5 = **7.85 点**
- 素点: 90.0 + 7.85 = **97.85 点 / 109 点**
- 100 点換算: 97.85 × 100 / 109 = **89.8 点 / 100 点**
- 判定: **A**

**例5: 製品レビュー・購入レポート（106点満点、軽量化モード）**

軽量化モード配点（SoT 表参照）: F=2.0 / Q=1.0 / C=0.5 / R=1.0 / A=1.0 / O=0.5 / G=N/A → 第11項目満点 6.0 点、記事全体満点 **106 点**

- 第1〜10項目: 各 9.0 点 × 10 = 90.0 点（第10項目で COI 開示済み）
- 第11項目（軽量化モード）:
  - F = 2.0 − 0.4（F-3 チェリーピッキング懸念、Must 欠落だが軽量化で減点率 20% 適用）= **1.6 点**
  - Q = 1.0（軽量化で使用条件明記、満点）
  - C = 0.5（軽量化で評価基準明示、満点）
  - R = 1.0（軽量化で使用環境明記、満点）
  - A = 1.0（軽量化、所感ベースで OK）
  - O = 0.5（公式仕様リンクあり）
  - G = N/A
  - サブ計（取得）: 1.6 + 1.0 + 0.5 + 1.0 + 1.0 + 0.5 = **5.6 点 / 6.0 点**
- 素点: 90.0 + 5.6 = **95.6 点 / 106 点**
- 100 点換算: 95.6 × 100 / 106 = **90.2 点 / 100 点**
- 判定: **S**（99〜100 点換算で S）

> 軽量化モードでは Must 欠落の減点率を 50% → 20% に緩和（Nice と同率扱い、軽量化前提のため）。設定欄「記事タイプ」が軽量化モード対象（製品レビュー等）の場合に自動適用。レビュー結果フォーマットの冒頭で「軽量化モード採用」と明示すること。

## 独立エージェントによる多段レビュー（推奨運用）

単一のレビューだけでは検出できない「構成の違和感」「グラフの誤誘導」「事実との乖離」「文脈の不整合」等は、**目的別の独立エージェントを複数スポーンして多角的に検証**する。**1 記事を 1 度で完璧にレビューできることは稀**であり、段階的改善 + Phase 間のエージェント検証を前提にする。

### 基本パターン: Phase 型多段レビュー

| Phase | 目的 | 想定範囲 |
|---|---|---|
| Phase 1 | 最小修正: 明らかな誤り（データ誤記、編集メタ残存、章番号ズレ） | 局所修正 |
| Phase 2 | 構成再編: 章分割、章番号シフト、関連ファイル同期 | 大規模リファクタ |
| Phase 3 | 冗長削減: 重複注記、長い所感、付録・本文の重複 | 全体見直し |
| Phase 4 | 精度検証: グラフ数値、本文と一次ソースの一致、表現の妥当性 | データ総点検 |
| Phase 4b | **公式反論・否認検索（4-gate 対応）**: 外部主体名 × 断定キーワードを検出したら必須実行。当事者の公式反論を Web 検索し両論併記の有無を検証 | ゲート判定 |
| Phase 5 | プラットフォーム適合: 0-gate（絵文字・タグ制限等）、SEO、画像最適化、リンク確認 | 公開準備 |

各 Phase の修正後に次のエージェントレビューを回し、スコア推移をメタ記録する（例: 68 → 76 → 88 → 92）。

### スコアリング付きレビューの依頼テンプレート

独立エージェントを呼び出す際は、**評価観点 + スコア制 + 残課題 + 公開判断** の 4 段階で依頼する。

```

認定ソース: <一次データファイル・公式ドキュメント等>

評価観点（各 10 点、合計 100 点）:
1. 章構成の論理性
2. 冒頭サマリの圧縮度
3. 図表配置の適切性
4. 冗長削減
5. 用語導線
6. 事実/所感分離
7. ナビゲーション
8. 文体一貫性
9. 想定読者への配慮
10. 公開判断（Yes/No）

出力（日本語, 400 字以内）:
1. 最終スコア + 判定 (S/A/B/C/D)
2. 前 Phase からの改善点 TOP 3
3. 残課題（あれば 2 項目以内）
```

### 公式反論・否認検索レビュー（Phase 4b, 4-gate 対応, **外部主体の断定的記述がある記事で必須**）

> **必須実行条件**: 記事に以下のいずれかがある場合、公開前に **必ず 1 回実行** する（実施ログは作業用ディレクトリに保存し、コミットメッセージ or 公開記録に実施有無を明記）。
>
> - 外部企業・組織・著名人の **固有名詞** を含む
> - 「赤字」「黒字」「損失」「利益」「採算性」「持続不可能」「ロスリーダー」「失敗」「成功」「最速」「最低」のいずれかのキーワードを含む
> - 「複数の報道によれば」「〜と報じられている」「業界では〜と言われている」「内部関係者によれば」の曖昧主語を含む
> - 外部主体の戦略判断・性能・評判を断定（「〜社は〜と判断した」「〜社は〜を狙っている」「〜は使い物にならない」）
>
> **実行せずに記事を公開した場合、事後的に 4-gate 違反として D 判定・撤去対象**。

外部企業・組織・個人について、**セカンダリソース（報道・解説記事・ブログ・SNS の伝聞）** だけを根拠に断定している主張がある場合、**当事者の公式反論・否認・訂正声明が存在するか** を独立エージェントに検索させる。報道後に当事者が SNS（X / LinkedIn / Bluesky 等）・公式ブログ・カンファレンス登壇・IR 資料・プレスリリースで否認しているケースが散見される。セカンダリソースの孫引きが一次情報のように扱われる失敗パターンを防ぐため、このレビューを **記事公開前に 1 回は必ず実施** する。

#### 依頼テンプレート

```
対象ファイル:
- <記事 MD / HTML>

検証対象の主張（抽出手順）:
1. 記事内で「特定主体（企業・組織・個人）の財務、採算性、戦略判断、製品性能、評判」に関する断定的主張を全て列挙
2. 各主張について、典拠が一次ソース（当事者発言・公式ドキュメント・決算資料・論文）かセカンダリソース（報道・解説記事・ブログ）かを分類

必須検証項目:
1. セカンダリソースのみを典拠とする主張について、当事者（CEO・CFO・広報・IR・本人等）の公式反論・否認・訂正声明が公開の場（SNS、ブログ、カンファレンス登壇、IR 資料、プレスリリース、決算資料）に存在するか Web 検索
2. 反論が存在する場合、本文で両論併記されているか
3. 反論の一次ソース（当事者本人の発言）と孫引きセカンダリソース（メディアが報じた反論）を区別しているか
4. 過去時点の主張を現在の文脈で引用する場合、時点の違いが明記されているか

検索クエリの例:
- "<人物名> <製品名> <数値> denied"
- "<人物名> <報道社> response"
- "<製品名> <財務指標> official statement"
- 日本語: "<人物名> <製品名> 反論" "<企業名> <報道社> 否認"

出力:
1. 各主張の典拠分類表（一次/セカンダリ、当事者反論有無）
2. 反論が存在するのに本文で言及されていない主張の行番号リスト
3. 推奨修正案（両論併記の雛形、反論ソースの追加脚注案）
4. 最終公開判断（反論未反映の断定がある場合は Hold、全て両論併記済みなら OK）
```

#### 判定基準

- **Hold（公開不可）**: セカンダリソースだけを根拠に断定している主張について、当事者の公開反論が存在するにも関わらず本文で言及されていない。
- **条件付き OK**: 当事者反論の Web 検索を実施済みで、見つかった反論は両論併記、見つからなかった場合は「セカンダリソース由来である旨」を脚注で明記。
- **OK**: 当事者発言の一次ソースのみを根拠とする主張、または数値が公式 IR 資料・決算資料・論文で裏取り済み。

### エージェントレビュー運用の原則

1. **レビュワーは元ファイルを編集しない**（改善案の提案にとどめる、修正は執筆者）
2. **スコア推移をメタ記録**（Phase 1: 68 → Phase 2: 76 → Phase 3: 90 等、改善可視化）
3. **A/B 比較は独立エージェントに任せる**（評価バイアスを避けるため、提案者自身ではなく別エージェントに比較を依頼）
4. **一次ソース突合は最優先**（Phase ごとに 1 回は必ず実施、数値ドリフト防止）
5. **残課題を次 Phase に持ち越す**（A 判定到達後も軽微な課題は改訂履歴に残す）
6. **複数 Phase を 1 commit にまとめない**（Phase ごとに保存、スコア遷移を履歴で追える）
7. **Phase 4b（公式反論検索）は外部主体記述がある記事で必須**。スキップは 4-gate 違反として事後的に判定 D・撤去対象。実施ログを残す。
8. **Phase 5（プラットフォーム適合性）は公開直前に必須**。0-gate 違反の事前検出。
