# 変更レビュー & テスト最適化 — 詳細基準

> **対象**: 任意のプロジェクトの最新変更（git diff / 特定ファイル）
> **目的**: 実装の正確性とテストの網羅性を客観的に100点満点で評価し、プロジェクト固有のテスト最適化提案を行う。
> **実行**: `/review-changes` スラッシュコマンド（グローバルスキル）

---

## コンセプト

### 既存スキルとの違い

| スキル | 対象 | 評価軸 | 粒度 |
|--------|------|--------|------|
| `/review-implementation` | プロジェクト全体 | テスト・正確性・マネタイズ・ペルソナ・UX (5軸) | 広範囲・総合 |
| `/review-changes` | **最新の変更差分** | **実装正確性・テストカバレッジ・テスト品質・追跡可能性 (4軸)** | **変更に特化・深掘り** |
| `three-agent/reviewer.md` | イテレーション成果物 | TDD証跡・カバレッジ・実装整合 (6軸) | three-agent専用 |

### 設計思想

1. **エージェント分離**: `/review-changes` は必ず別エージェント（別セッション）で実行される。実装コンテキストを遮断し客観性を確保する（詳細は `agents.md` 参照）
2. **プロジェクト自動適応**: テストフレームワーク・設定・パターンを自動検出し、プロジェクト固有の基準で評価する
3. **半自動運用**: `/review-changes` だけで実行可能。引数なしで直近の変更を自動取得する
4. **証拠主義**: 推測でスコアを付けない。各スコアにはファイルパスと行番号の引用が必須。証拠不十分時は各レンジの下限で採点する
5. **テスト最適化**: レビューに加え、プロジェクト固有のテスト改善提案を出力する

### 配点設計の理由

変更レビューではテスト関連（軸2+軸3=55点）に重点を置く。`/review-implementation` にある マネタイズ・ペルソナ・UX は変更スコープ外のため除外している。

---

## 評価軸の詳細

### 軸1: 実装の正確性 (30点)

変更されたコードが正しく動作するかを、コードの事実のみに基づいて評価する。

**重点チェック項目**:
- ロジックの論理的正確性（条件分岐、ループ、計算）
- エッジケース対応（null, 空, 境界値, 並行処理）
- エラーハンドリング（外部依存失敗、データ不整合防止）
- セキュリティ（インジェクション、バリデーション、情報漏洩）
- 依存関係の変更（package.json等）がある場合、バージョン互換性・脆弱性を確認
- パフォーマンスに影響する変更（アルゴリズム変更・N+1クエリ等）がある場合、影響を評価

**配点ガイド**:
- 27-30: ロジック正確、エッジケース対応済、セキュリティ問題なし
- 21-26: おおむね正確だが一部不足
- 15-20: 正常系は動くが異常系に問題
- 0-14: 重大な論理エラーまたはセキュリティ問題

### 軸2: テストカバレッジの網羅性 (30点)

変更に対するテストが十分に書かれているかを評価する。

**重点チェック項目**:
- 新規/変更された関数にテストが存在するか
- 正常系・異常系・境界値・組み合わせの網羅
- 副作用（API/DB/ファイル）のモック分離
- 回帰テスト（バグ修正には再発防止テスト）

**配点ガイド**:
- 27-30: 全変更にテストあり、正常/異常/境界値を網羅
- 21-26: 主要な変更にテストあるが一部不足
- 15-20: 正常系テストのみ、大きなカバレッジ漏れ
- 0-14: テストがほぼない

### 軸3: テスト品質・戦略 (25点)

テスト自体の品質とプロジェクトのテスト戦略を評価する。

**重点チェック項目**:
- フレーキーテストの有無
- アサーションの厳密さ（甘いアサーションの検出）
- テスト設計（命名、AAAパターン、データ管理）
- ハードコード/ごまかしの検出
- テスト戦略の適切さ（テストピラミッド、CI統合）

**配点ガイド**:
- 22-25: 高品質、戦略適切、ごまかしなし
- 17-21: おおむね良好だが一部問題あり
- 12-16: アサーションが甘い、戦略不明確
- 0-11: 品質が低い、ごまかしあり

### 軸4: 変更の追跡可能性 (15点)

変更の意図・影響範囲が追跡可能かを評価する。

**重点チェック項目**:
- コミットメッセージの質
- ドキュメントの整合性
- 影響範囲の可視性
- 破壊的変更のマイグレーション手順

**配点ガイド**:
- 13-15: 変更意図が明確、ドキュメント整合
- 10-12: おおむね追跡可能だが一部漏れ
- 6-9: コミットメッセージが不明瞭
- 0-5: 追跡不可能

## テスト最適化提案の観点

レビュー結果に基づき、以下の観点でプロジェクト固有の提案を行う。

### 1. テスト種別の充足度
- Unit / Integration / E2E のバランスをテストピラミッドで評価
- プロジェクトの性質（API / フロントエンド / CLI / ライブラリ）に応じた推奨比率

### 2. テストインフラ改善
- カバレッジ計測の導入/改善
- CI/CDでのテスト自動実行
- テスト並列化による実行速度改善
- スナップショットテスト/ビジュアルリグレッションの検討

### 3. テストパターン改善
- フィクスチャ/ファクトリの整理
- 共通ヘルパー・カスタムマッチャーの導入
- テストデータのシード管理

### 4. 具体的テストケース提案
- 変更に対して**具体的なテストコード**（擬似コードでも可）を提案
- テスト名・入力値・期待結果を明示

## ハードコード/ごまかし検出パターン

### ハードコード実装の例
```python
# NG: テストケースの値をそのまま返している
def calculate_tax(amount):
    if amount == 1000:
        return 100  # テストケースをハードコード
    return 0

# OK: 正しいロジック
def calculate_tax(amount):
    if amount <= 0:
        return 0
    return amount * 0.1
```

### テストごまかしの例
```python
# NG: アサーションが甘い
def test_calculation():
    result = buggy_function(100)
    assert result > 0  # 本来は == 110

# OK: 厳密なアサーション
def test_calculation():
    result = correct_function(100)
    assert result == 110
```

### skip/pendingの濫用
```javascript
// NG: 失敗するテストをskipで回避
it.skip('should handle edge case', () => { ... })

// OK: 実装予定としてtodoを使い、理由を明記
it.todo('should handle edge case - blocked by #123')
```

## 運用ガイド

### 推奨タイミング

| タイミング | 使い方 |
|-----------|--------|
| PR作成前 | `/review-changes` で自己チェック |
| コードレビュー時 | 別のClaude Codeセッション（別エージェント）で `/review-changes` を実行して客観的レビュー |
| バグ修正後 | 修正の正確性と再発防止テストを確認 |
| リファクタリング後 | 既存テストが壊れていないか、カバレッジが維持されているか確認 |

### エージェント分離によるレビュー（推奨）

**`/review-changes` は必ず別エージェントで実行される。** 詳細は `agents.md` 参照。

**自動分離（推奨）:**
```
実装セッションで /review-changes を実行
→ Agent ツールが自動で別エージェントを起動
→ コード差分とテスト設定のみが渡される（実装コンテキストは遮断）
→ レビュー結果が返却される
```

**手動分離:**
```
ターミナル1 (実装者): コードを書く・テストを書く
ターミナル2 (レビュアー): 新しい Claude Code セッションを開き、
  docs/review-changes.md を読み込ませてレビューを依頼
```

### 静的分析の制約

本スキルはコード実行を行わないため、テストのパス/フェイルは静的分析のみで判断する。
CI結果やテスト実行ログが利用可能な場合は、その情報も参照すること。

### テスト基盤が未検出の場合

テスト基盤が検出できないプロジェクトでは、軸2・軸3は「テスト未導入のため0点」とし、テスト導入を最優先提案として出力する。
