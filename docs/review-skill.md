# Skills 品質レビュープロンプト

> **対象**: Claude Code スラッシュコマンド（`.claude/commands/*.md`）および Claude.ai Skills（SKILL.md）
> **目的**: 作成済みの Skill / コマンドが Anthropic のベストプラクティスに沿っているか、5軸で100点満点評価する。
> **参照マニュアル**: `docs/skills-building-guide.md`（Anthropic公式 "The Complete Guide to Building Skills for Claude" に基づく）

---

## レビュー前の必須確認事項

レビューを開始する前に、以下をスキルファイルから確認すること。

| 確認項目 | 確認内容 |
|----------|----------|
| **スキルの形式** | Claude Code コマンド（`.claude/commands/*.md`）か SKILL.md 形式か |
| **対象ユースケース** | どのカテゴリか（Document Creation / Workflow Automation / MCP Enhancement） |
| **評価基準ファイルの有無** | `docs/review-*.md` 等の詳細基準ファイルがあるか |
| **スキル作成マニュアル** | `docs/skills-building-guide.md` を読み込んで評価基準として使用する |

---

## レビュー観点（5項目・各20点）

---

### 1. 構造・命名・フォーマット 【配点：20点】

> **目的**: スキルがAnthropicの技術要件に準拠しているかを確認する。

#### ファイル・フォルダ構造
- [ ] ファイル名が正しい（SKILL.md 形式の場合は正確な綴り、コマンド形式の場合は kebab-case.md）
- [ ] フォルダ名が kebab-case（SKILL.md 形式の場合）
- [ ] スキルフォルダ内に不要な README.md がない（SKILL.md 形式の場合）
- [ ] 詳細な評価基準が別ファイル（`docs/review-*.md` 等）に分離されているか（Progressive Disclosure）

#### YAML フロントマター / ヘッダー
- [ ] Claude Code コマンドの場合: 冒頭に明確なタイトルと説明がある
- [ ] SKILL.md 形式の場合: `---` デリミタで囲まれた YAML フロントマターがある
- [ ] name フィールドが kebab-case（SKILL.md 形式の場合）
- [ ] XML タグ（`<` `>`）がフロントマターに含まれていない
- [ ] スキル名に `claude` / `anthropic` が含まれていない

#### Markdown 品質
- [ ] 見出し階層が論理的か（`#` → `##` → `###`）
- [ ] コードブロック・テーブル・リストが正しくフォーマットされているか
- [ ] 全体が5,000語以内に収まっているか（超える場合は references/ に分離すべき）

**採点目安**

| 点数 | 基準 |
|------|------|
| 18〜20 | 全技術要件を満たし、Progressive Disclosure が適切に実装されている |
| 14〜17 | おおむね準拠だが、分離やフォーマットに軽微な問題あり |
| 10〜13 | 基本構造はあるが命名規則違反や構造上の問題がある |
| 0〜9  | 技術要件を大幅に満たしていない |

---

### 2. Description / トリガー設計 【配点：20点】

> **目的**: スキルが適切なタイミングで発火し、不要な場面では発火しないかを確認する。

#### 「何をするか」の記述
- [ ] スキルの目的が1-2文で明確に説明されているか
- [ ] 具体的な成果物・アウトプットが分かるか
- [ ] 対象カテゴリ（Document Creation / Workflow Automation / MCP Enhancement）が推測できるか

#### 「いつ使うか」の記述
- [ ] ユーザーが実際に言いそうなトリガーフレーズが含まれているか
- [ ] 言い換え・パラフレーズでもトリガーされるよう複数のフレーズがあるか
- [ ] 関連するファイルタイプが記載されているか（該当する場合）

#### トリガー品質
- [ ] description が曖昧すぎないか（「Helps with projects」のような表現がないか）
- [ ] description が広すぎないか（無関係なクエリでも発火しそうにないか）
- [ ] ネガティブトリガー（使うべきでない場面）が明記されているか（必要な場合）
- [ ] 他のスキルとの責任境界が明確か

**採点目安**

| 点数 | 基準 |
|------|------|
| 18〜20 | 「何を・いつ」が明確、トリガーフレーズが豊富、境界が明確 |
| 14〜17 | 概ね良いが、トリガーフレーズが不足、またはやや曖昧 |
| 10〜13 | 目的は分かるがトリガー設計が弱い |
| 0〜9  | 何をするスキルか分からない、またはトリガー条件がない |

---

### 3. 命令の品質・明確性 【配点：20点】

> **目的**: Claude がスキルの命令に従って一貫した結果を出せるかを確認する。

#### 具体性・アクション可能性
- [ ] 各ステップが具体的で実行可能か（「適切に処理する」のような曖昧表現がないか）
- [ ] 必要なパラメータ・入力値が明記されているか
- [ ] 成功・失敗の判定基準が示されているか
- [ ] コマンド・スクリプトの呼び出しが正確か

#### ステップ構造
- [ ] ステップが論理的な順序で並んでいるか
- [ ] ステップ間の依存関係が明確か
- [ ] バリデーションゲート（次に進む前の確認）があるか
- [ ] 重要な命令が冒頭に配置されているか（`## Important` / `## Critical` 等）

#### 例・ユースケース
- [ ] 典型的なユースケースの例が含まれているか
- [ ] ユーザーの発言 → アクション → 結果の流れが示されているか
- [ ] エッジケースへの対応が示されているか

#### エラーハンドリング
- [ ] 一般的なエラーとその対処法が記載されているか
- [ ] MCP 接続エラー等の外部依存エラーへの対処があるか（該当する場合）
- [ ] ロールバック・フォールバック手順があるか（該当する場合）

**採点目安**

| 点数 | 基準 |
|------|------|
| 18〜20 | 具体的でアクション可能、例が豊富、エラーハンドリング完備 |
| 14〜17 | おおむね良いが、一部の命令が曖昧、または例・エラー対処が不足 |
| 10〜13 | 基本的な手順はあるが、曖昧な表現が多い、または例がない |
| 0〜9  | 命令が抽象的すぎて Claude が一貫した結果を出せない |

---

### 4. 出力設計・一貫性 【配点：20点】

> **目的**: スキルの出力が構造化され、一貫性のある高品質な結果を生むかを確認する。

#### 出力フォーマット
- [ ] 出力テンプレート・フォーマットが定義されているか
- [ ] 採点基準がある場合、採点表・判定表が含まれているか
- [ ] 出力に含めるべきセクション・フィールドが明確か
- [ ] 不要な情報を含めない制約が示されているか

#### 一貫性の担保
- [ ] 同じ入力に対して毎回同じ構造の出力が得られるか
- [ ] 評価軸がある場合、各軸の配点・採点目安が明確か
- [ ] ユーザーが追加の指示をしなくてもワークフローが完遂するか

#### 行動原則・制約
- [ ] スキルの行動原則（証拠主義・非破壊等）が明記されているか
- [ ] やってはいけないこと（コード編集・実行禁止等）が明記されているか
- [ ] 判断基準の根拠（ファイルパス・行番号の引用等）が求められているか

**採点目安**

| 点数 | 基準 |
|------|------|
| 18〜20 | 出力テンプレートが明確、行動原則が確立、高い再現性 |
| 14〜17 | テンプレートはあるが一部不明確、または制約が不十分 |
| 10〜13 | 出力形式の指定が弱い、結果にばらつきが出やすい |
| 0〜9  | 出力設計がなく、毎回異なる形式の結果になる |

---

### 5. 実用性・保守性 【配点：20点】

> **目的**: スキルが実際に役立ち、長期間にわたって保守・改善できるかを確認する。

#### 実用性
- [ ] 想定ユーザーが実際に使うシナリオが明確か
- [ ] 前提条件（必要なファイル・設定・MCP接続等）が明記されているか
- [ ] スキルなしの場合と比べて明確な改善があるか（やりとり回数削減・品質向上等）
- [ ] 他のスキルとの組み合わせが考慮されているか（Composability）

#### 保守性
- [ ] 別の開発者がスキルを読んで理解・修正できるか
- [ ] バージョン情報・更新履歴が管理されているか（metadata等）
- [ ] CLAUDE.md や README 等のドキュメントに反映されているか
- [ ] 詳細基準が別ファイルに分離され、独立して更新可能か

#### ポータビリティ
- [ ] プロジェクト固有のパスがハードコードされていないか
- [ ] 他のプロジェクトにコピーしても動作するか
- [ ] 環境依存（OS・ツール・MCP）が明記されているか

**採点目安**

| 点数 | 基準 |
|------|------|
| 18〜20 | 明確なユースケース、高い保守性、ポータブル、ドキュメント完備 |
| 14〜17 | 実用的だが、保守性やドキュメントに軽微な不足 |
| 10〜13 | 動くが、保守や他プロジェクト展開に課題あり |
| 0〜9  | 実用性が低い、または保守が困難 |

## 採点方法

### 総合点：100点満点

| 評価 | 点数 | 判定 |
|------|------|------|
| S | 90〜100点 | 高品質。そのまま配布・共有可能 |
| A | 80〜89点 | 軽微な改善後に配布可能 |
| B | 70〜79点 | 重点項目の改善が必要。修正後に再レビュー |
| C | 60〜69点 | 大幅な改善が必要。設計見直しを検討 |
| D | 59点以下 | 根本的な再設計が必要 |

## レビュー実施ガイド

### このプロンプトの使い方

**Claude Code での使い方（スラッシュコマンド）:**
```
/review-skill                          ← リポジトリ内の全スキルをレビュー
/review-skill .claude/commands/xxx.md  ← 特定スキルをレビュー
```

**手動で使う場合:**
1. このプロンプトを Claude Code セッションに貼り付ける
2. 「このスキルをレビューしてください」と依頼する
3. 対象のスキルファイルを指定する

### レビュータイミングの推奨

| タイミング | 目的 |
|----------|------|
| スキル新規作成後 | Anthropic ベストプラクティスへの準拠確認 |
| スキル改修後 | 改修が品質を維持しているか確認 |
| 他プロジェクト展開前 | ポータビリティの確認 |
| 定期レビュー | スキル群全体の品質維持 |
