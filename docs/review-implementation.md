# 実装品質 総合レビュープロンプト

> **対象**: Claude Code で開発中・完成したプロジェクトの実装成果物
> **目的**: 「テスト品質」「処理の正確性」「マネタイズ設計整合性」「ペルソナ適合実装」「UX品質」の5軸で実装を100点満点で評価する。

---

## レビュー前の必須確認事項

レビューを開始する前に、以下をプロジェクト文書または開発者から確認すること。

| 確認項目 | 確認内容 |
|----------|----------|
| **このプロダクトは何のためにあるか？** | 解決する課題を1〜2文で言語化できるか |
| **誰が使うか？（ペルソナ）** | `dev/design/persona.md` に基づく対象ユーザーが定義されているか |
| **どうやって収益化するか？** | 収益モデル・価格設定が明文化されているか |
| **テスト戦略は何か？** | TDD / BDD / E2E など、どのテスト方針で開発したか |

上記が不明瞭な場合は、レビュー開始前にプロジェクト文書を確認すること。

---

## レビュー観点（5項目・各20点）

---

### 1. テスト実装品質 【配点：20点】

> **目的：** テストが実装の信頼性を担保しており、TDD原則に沿って書かれているかを確認する。

#### テストファースト確認
- [ ] テストが実装より先に書かれた証跡があるか（失敗ログ → 成功ログの流れ）
- [ ] テストが「仕様の文書」として機能しているか（テスト名を読めば要件が分かる）

#### カバレッジ確認
- [ ] **正常系**（ハッピーパス）が網羅されているか
- [ ] **異常系**（エラーハンドリング・バリデーション失敗）がテストされているか
- [ ] **境界値**（ゼロ・空文字・最大値・最小値）がテストされているか
- [ ] **副作用**（DB操作・API呼び出し・ファイルI/O）がモック/スタブで分離されているか

#### テスト品質確認
- [ ] テストがフレーキー（実行タイミングで結果が変わる）でないか
- [ ] テスト間で状態が共有・汚染されていないか
- [ ] ハードコードされた値や「テストのためのごまかし」がないか

**採点目安**

| 点数 | 基準 |
|------|------|
| 18〜20 | TDD証跡あり、正常系・異常系・境界値が網羅、テスト品質が高い |
| 14〜17 | おおむね網羅されているが、一部の境界値や副作用テストが不足 |
| 10〜13 | テストはあるが正常系のみ、または品質が低い（ハードコード等） |
| 0〜9  | テストがほぼなく、TDDの原則が守られていない |

---

### 2. 処理の正確性 【配点：20点】

> **目的：** 実装が仕様・要件通りに正確に動作し、エッジケースや異常系でも適切に振る舞うかを確認する。

#### ビジネスロジック確認
- [ ] 計算・変換ロジックが仕様と一致しているか（手計算で検証可能か）
- [ ] 条件分岐が漏れなく実装されているか（全パターンが網羅されているか）
- [ ] データの型・フォーマット・単位が一貫しているか

#### エラーハンドリング確認
- [ ] 想定される例外・エラーがすべてキャッチされているか
- [ ] エラー発生時のユーザーへのフィードバックが適切か
- [ ] 失敗時にデータが破損・消失しない安全な処理になっているか
- [ ] 外部API・DB接続失敗時のフォールバック処理があるか

#### セキュリティ・データ整合性
- [ ] SQLインジェクション・XSS・コマンドインジェクションなどの脆弱性がないか
- [ ] 入力値のバリデーションがシステム境界（ユーザー入力・外部API）で行われているか
- [ ] センシティブな情報（パスワード・APIキー）が平文でログ出力されていないか

**採点目安**

| 点数 | 基準 |
|------|------|
| 18〜20 | ロジックが正確、エラーハンドリング完備、セキュリティ問題なし |
| 14〜17 | おおむね正確だが、一部のエッジケースやエラー処理が不足 |
| 10〜13 | 正常系は動くが、異常系で予期しない挙動が発生しうる |
| 0〜9  | ロジックに重大な誤りがある、またはセキュリティ上の問題がある |

---

### 3. マネタイズ設計整合性 【配点：20点】

> **目的：** 実装がプロダクトの収益モデルを正確に反映しており、マネタイズを阻害する実装上の問題がないかを確認する。

#### 収益モデルとの整合確認
- [ ] 課金ロジック（サブスクリプション・従量課金・一括購入等）が正確に実装されているか
- [ ] 無料プランと有料プランの境界が明確にコード上で表現されているか
- [ ] 試用期間・クーポン・割引ロジックが仕様通りに動作するか

#### 収益に影響するUX確認
- [ ] アップグレード導線（無料 → 有料）がユーザーに自然に提示されているか
- [ ] 決済フロー（エラー・リトライ・確認画面）がユーザー離脱を防ぐ設計になっているか
- [ ] 支払い失敗時の処理が適切か（アカウント停止タイミング・再試行フロー）

#### 分析・計測
- [ ] 収益に直結するKPI（コンバージョン・解約・LTV）が計測可能な実装になっているか
- [ ] A/Bテストや価格テストが実施できる設計になっているか
- [ ] 不正利用・悪用を防ぐ仕組みがあるか（レート制限・使用量監視等）

**採点目安**

| 点数 | 基準 |
|------|------|
| 18〜20 | 収益モデルが正確に実装、アップグレード導線が自然、計測体制が整っている |
| 14〜17 | 基本的な課金ロジックは正しいが、導線や計測の一部が不足 |
| 10〜13 | 課金ロジックはあるが、マネタイズ最適化が不十分 |
| 0〜9  | 収益モデルが実装に反映されていないか、重大なバグがある |

---

### 4. ペルソナ・クライアント適合実装 【配点：20点】

> **目的：** 実装が `dev/design/persona.md` で定義したペルソナの実際の使い方・ITリテラシー・使用シーンに適合しているかを確認する。

#### ペルソナ定義との整合確認
- [ ] ペルソナのITリテラシーに合わせた複雑さになっているか（高ITリテラシー向けに作ったものを低リテラシー向けに提供していないか）
- [ ] 主な使用デバイス（スマホ中心 / PC中心）に対応したレイアウト・操作になっているか
- [ ] 使用シーン（通勤中・オフィス・外出先等）に適したUI・フロー設計になっているか

#### ペインポイント解消確認
- [ ] ペルソナが「最も困っていること」を直接解決できる機能が最短で使えるか
- [ ] ペルソナが「このサービスを選ぶ理由」となる差別化機能が実際に動作しているか
- [ ] ペルソナが離脱しそうなポイント（複雑な操作・理解しにくい画面）がないか

#### 文言・ラベル確認
- [ ] 画面上のラベル・ボタン・メッセージがペルソナの語彙・文化に合っているか
- [ ] エラーメッセージがペルソナに理解できる言語で書かれているか（専門用語のない説明）
- [ ] ヘルプ・ガイダンスがペルソナのITリテラシーに合わせて提供されているか

**採点目安**

| 点数 | 基準 |
|------|------|
| 18〜20 | ペルソナ定義に完全に適合、ペインポイントが直接解消され、文言も適切 |
| 14〜17 | おおむね適合しているが、一部でペルソナとのズレがある |
| 10〜13 | 基本機能は動くが、ペルソナの実際の使い方に合っていない部分がある |
| 0〜9  | ペルソナ分析が実装に反映されておらず、誰のための実装か不明 |

---

### 5. UX品質 【配点：20点】

> **目的：** `dev/design/` のデザインシステム・ガイドラインに基づき、ユーザーが迷わず目的を達成できる品質になっているかを確認する。

#### インタラクション品質
- [ ] ユーザーの操作に対してフィードバック（ローディング・成功・エラー）が即時に返るか
- [ ] 待機時間が発生する処理にプログレス表示があるか
- [ ] 操作の取り消し（Undo）・確認ダイアログが破壊的操作に対して実装されているか

#### ビジュアル・レイアウト品質
- [ ] `dev/design/design-guidelines.md` の禁止事項（絵文字禁止等）が守られているか
- [ ] IBM Carbon Design System のデザイントークン・コンポーネントが正しく使われているか
- [ ] レスポンシブ対応が適切か（モバイル・タブレット・デスクトップ）
- [ ] ダークモード対応が必要な場合に実装されているか

#### アクセシビリティ・パフォーマンス
- [ ] キーボード操作のみで主要タスクを完了できるか
- [ ] 画像・アイコンにalt属性・aria-labelが付いているか
- [ ] 初期表示が3秒以内に完了するか（Lighthouseスコア等で確認）
- [ ] 操作のたびに全体リロードが発生しない設計になっているか

**採点目安**

| 点数 | 基準 |
|------|------|
| 18〜20 | インタラクション・ビジュアル・アクセシビリティすべてが高品質 |
| 14〜17 | おおむね良好だが、一部のフィードバックやアクセシビリティが不足 |
| 10〜13 | 動作はするがUXが粗く、ユーザーが迷う場面がある |
| 0〜9  | UXの基本が実装されておらず、ユーザーが使いこなせない |

## 採点方法

### 総合点：100点満点

| 評価 | 点数 | 判定 |
|------|------|------|
| S | 90〜100点 | リリース可。軽微な改善は次イテレーションで対応 |
| A | 80〜89点 | 軽微な修正後にリリース可 |
| B | 70〜79点 | 重点項目の修正が必要。修正後に再レビュー |
| C | 60〜69点 | 大幅な修正が必要。設計見直しを検討 |
| D | 59点以下 | 根本的な再設計が必要 |

## レビュー実施ガイド

### このプロンプトの使い方

**Claude Code での使い方（スラッシュコマンド）:**
```
/review-implementation
```

**手動で使う場合:**
1. このプロンプトをClaude Codeセッションに貼り付ける
2. 「このプロジェクトの実装をレビューしてください」と依頼する
3. 必要に応じて以下の情報を提供する：
   - `dev/design/persona.md` の内容
   - 収益モデル・マネタイズ方針
   - テスト戦略

### レビュータイミングの推奨

| タイミング | 目的 |
|----------|------|
| 各イテレーション完了後 | `dev/three-agent/reviewer.md` と併用して品質を確認 |
| リリース前 | 総合評価でS/A判定を確認 |
| 重大バグ修正後 | 修正が他の観点に悪影響を与えていないか確認 |
| ペルソナ変更後 | 実装がペルソナ変更を反映しているか確認 |
